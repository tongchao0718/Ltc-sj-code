# power-fee-protocol-check 数据库设计文档

## 1. 设计范围与目标

- 覆盖模块：`M01~M14`（模板、样本、规则、任务、解析、校核、问题治理、监控、优化、复核）。
- 设计目标：保证协议核查链路“可追溯、可审计、可恢复、可统计”。
- 统一主键建议：业务主表使用 `varchar(64)` 业务ID（如 `task_id`、`issue_id`），并保留自增 `id` 作为物理主键（可选）。

## 3. 关系设计（关键）

1. `protocol_template` 1:N `protocol_template_field`  
2. `protocol_rule` 1:N `rule_version`；`rule_version` 1:N `rule_execute_log`  
3. `task_strategy` 1:N `protocol_task`；`protocol_task` 1:N `protocol_task_detail`  
4. `protocol_parse_result` 1:N `verify_item_detail`（通过 `verify_id` 关联）  
5. `protocol_problem` 1:N `governance_ticket`（同问题可多轮治理）  
6. `review_feedback` 1:N `manual_review_task`；复核结果可回写 `error_corpus`

## 4. 索引与性能建议

- 高频检索索引：
  - `protocol_task(task_status, created_at)`
  - `protocol_problem(issue_level, issue_status, org_code, created_at)`
  - `governance_ticket(ticket_status, assignee, updated_at)`
  - `task_monitor_log(task_id, error_type, created_at)`
- 唯一约束：
  - `task_id`、`issue_id`、`ticket_id`、`iteration_id`、`review_task_id`
- 分区建议（数据量大时）：
  - 日志类/报表类表按月分区（如 `rule_execute_log`、`task_monitor_log`）。

## 5. 数据一致性与审计要求

1. 规则发布必须写入 `rule_version` 并记录 `operator`、`updated_at`。  
2. 任务、校核、问题、工单全链路保留 `request_id/task_id/file_id` 可追溯字段。  
3. 状态变更需落日志表（如 `ticket_process_log`），不得仅改主表状态。  
4. 软删除字段建议统一：`is_deleted`、`deleted_at`、`deleted_by`。  

## 6. 开发落地注意事项

1. 枚举口径与 PRD 标准编码保持一致（任务状态、工单状态、规则类型）。  
2. 监控与报表相关表优先保证写入稳定，再逐步增强聚合查询。  
3. 字段变更需同步更新映射表、SDD JSON、验收脚本与版本变更记录。
