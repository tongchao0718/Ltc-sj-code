# 电费协议核查（power-fee-protocol-check）数据库设计文档

> 依据：《01-需求说明书》V2.1、《02-产品设计文档》、《03-PRD设计评审文档》、《04-页面-路由-接口-数据表映射表》、《01-详细设计文档》、服务端 `protocolDomainService.js`（演示态领域模型）及《核心文档/AI+产品落地/02-子应用通用模板/05-数据库设计文档模板.md》。

## 1. 设计口径

本子应用支持两种模式：

1. **演示模式**：当前服务端以进程内内存结构模拟领域实体（见 `AIEP-SERVER/.../protocolDomainService.js`），可无真实库表或仅部分落库。
2. **落地模式**：按本文档「物理模型清单」与接口契约对接真实数据库（建议 MySQL 8.x），表名与《04-页面-路由-接口-数据表映射表》一致。

原则：**演示可降级，评审可落地**。与 PRD/映射表冲突时，以评审结论为准，并回写本文档与 DDL。

## 4. 核心字段说明（可选）

字段级定义以第 7 章「物理模型清单」为准，本章不重复维护，避免双份不一致。

## 5. 数据一致性约束

- 状态类字段与标准编码对齐；落库值与接口枚举保持一致。
- **全链路追溯**：写入侧建议统一携带 `trace_id`（或网关 `request_id`），并在 `protocol_task`、`protocol_parse_result`、`protocol_verify_result`、`protocol_problem`、`governance_ticket`、关键日志表中可关联查询。
- **异步与批量**：任务生成、派发、复核提交等接口需幂等键（如 `idempotency_key`）可落扩展表或任务主表扩展字段。
- **同一协议文件**：解析 → 校核 → 问题治理使用统一 `file_id`（及关联 `parse_id` / `verify_id`）。
- **权限与审计**：敏感字段展示脱敏；状态迁移除主表外写入对应日志表（如 `ticket_process_log`）。

## 6. 变更管理

- 新增/变更库表须同步更新 PRD 物理模型摘要、《04-页面-路由-接口-数据表映射表》及接口契约。
- 字段冻结后变更走评审变更单；DDL 与第 7 章字段清单必须同源维护。
- 与 `protocolDomainService.js` 演示字段对齐时，以本文档落地字段为准，代码仅作演示子集映射。

### 6.1 关系与索引（实施建议）

1. `protocol_template` 1:N `protocol_template_field`；`protocol_template` N:M `protocol_baseline` 经 `template_baseline_mapping`。  
2. `protocol_rule` 1:N `rule_version`；执行侧 `rule_execute_log` 关联 `task_id` / `rule_code`。  
3. `task_strategy` 1:N `protocol_task`；`protocol_task` 1:N `protocol_task_detail`。  
4. `protocol_parse_result` 与 `protocol_verify_result` 建议以 `parse_id` 关联；`verify_item_detail` 关联 `verify_id`。  
5. `protocol_problem` 1:N `governance_ticket`；工单流转写入 `ticket_process_log`。  
6. 高频索引示例：`protocol_task(task_status, created_at)`、`protocol_problem(issue_status, org_code, created_at)`、`governance_ticket(ticket_status, assignee)`；日志类表可按月分区。
