# PRD 设计生成（prd-design-generation）

> **总编排**：`.cursor/skills/ai-full-process-design/SKILL.md`  
> **模板真值**：`核心文档/AI+产品落地/02-子应用通用模板/03-PRD设计评审文档模板.md`  
> **落盘路径**：`AIEP-WEB/src/docs/子应用文档/{app-code}/01-需求与设计/03-PRD设计评审文档.md`  
> **样例**：见 [examples.md](examples.md)

## 适用步骤（调用时必须指定模式）

| 模式 | 步骤 | 用途 |
|------|------|------|
| **step3-draft** | 3 界面生成 | 产出 `03-PRD` **草案**（与全量原型、04 并行） |
| **step4-freeze** | 4 界面确认 | G2-A 人工通过后升版 **冻结版** + 同步映射表 |
| — | 2 需求确认 | **禁止**使用本 Skill（写 SDD + 17 + 20-G2-B） |

## 双真值分工（强制）

| 内容 | 真值文档 | PRD 中怎么写 |
|------|----------|--------------|
| API 路径、字段、错误码、GWT | `SDD-v*.md` + `SDD-v*.json` | **引用** SDD 章节/条目，禁止字段级重复定义 |
| 页面布局、组件、四态、文案 | `04-界面设计文档.md` + 可交互原型 | PRD 写模块级需求与验收摘要；细节指向 04 |
| 模块需求、用户故事、评审清单 | **本文档 `03-PRD`** | 评审主文档 |
| 页面-路由-接口映射（Gate-3） | `页面-路由-接口-数据表映射表.md` | PRD §8.1 可写**摘要**；**独立文件为 Gate-3 真值** |

## 协作顺序

**步骤 3**（界面优先）：

```text
sub-app-resources → ui-generation（step3）→ 本 Skill（step3-draft）
  → 可选 prd-page-annotation → design-review-pre-g2a
```

**步骤 4**：

```text
ui-generation（step4）→ 本 Skill（step4-freeze）→ validate:sdd --gate G3
```

PRD 草案须在 **全量界面与 04 草案基本对齐** 后定稿，不得先于原型写「空想 PRD」。

## Workflow A：step3-draft（PRD 草案）

### 必要输入

- `01-需求说明书.md`（含 `platform_type`、`flow_type`、页面清单、SA-xx）
- `17-需求确认单.md`（PO+TL 双签）
- `SDD-v*.md`（草案或已签署）
- `ui-generation` step3 产出：全量可交互界面 + `04-界面设计文档.md` 草案
- 可选：`产品设计文档.md`

### 执行步骤

1. 从模板复制 `03-PRD设计评审文档模板.md` 到落盘路径（若不存在）
2. 填写「文档信息」：`文档状态` = `待评审` 或 `草案`；关联文档须含 `01-需求说明书.md`、`SDD-v*.md/json`
3. **一、项目概述**：对齐需求说明书范围与不做项
4. **二、系统流程**：业务流程 + 评审会前检查清单（2.4）
5. **三、用户故事**：US → SA → M → AT 追溯（3.0～3.4 详尽度检查全填）
6. **四、功能说明**：按模块 Mxx 六段式（需求/流程/设计/存储/注意/异常）；4.2 检查表全填
7. **五～七、** NFR、物理模型（引用 `数据库设计文档`）、标准编码
8. **八、附录**：8.1 映射摘要；8.2 **仅列接口名 +「详见 SDD §x」**；8.3 DoD 摘要
9. 与 `04`、代码路由核对：P0 页面 100% 有模块覆盖（支撑 U6、预审 D3）
10. 若已做 `prd-page-annotation`：浮窗需求合并进对应模块「（1）功能需求」

### step3-draft 完成标准（进入步骤 4 前）

| # | 条件 |
|---|------|
| P1 | 模板 **一～八章 + 附件** 结构齐全，禁止空章标题 |
| P2 | 每个 P0 模块「（1）功能需求」≥ 3 条 |
| P3 | 第三章 US → SA → M 追溯完整 |
| P4 | `文档状态` 非「冻结版」 |
| P5 | 与 U1～U6、`design-review-pre-g2a` D3 一致 |

未满足 → `fail`，退回修订；缺 PO 决策 → `blocked`。

### flow_type 分支

| flow_type | PRD 策略 |
|-----------|----------|
| `standard` | **全量**模块与用户故事；禁止 MVP/R1/R2 裁剪（与需求说明书一致） |
| `fast_track` | 可简化五～七章；须引用 `15-SDD-Lite`；用户故事仅覆盖受影响 SA |

## Workflow B：step4-freeze（PRD 冻结）

### 必要输入

- Workflow A 草案已完成
- **`19-G2-A界面确认记录.md` 人工「通过」**（PO/客户签字，非 Agent）
- 客户走查反馈已落实到界面与 04

### 执行步骤

1. **确认 G2-A**：Read `19-G2-A`；无人工通过 → 输出 **`blocked`**，**禁止**改 PRD 状态
2. 按走查反馈修订 PRD 对应模块（仅微调，禁止大规模新页面）
3. 更新版本变更记录；`文档状态` → **`冻结版`**，注明冻结日期与确认人
4. 完成/更新 `页面-路由-接口-数据表映射表.md`（Gate-3 真值，不可仅在 PRD 内维护）
5. 若有页面标注：最终浮窗内容合并进 PRD，避免研发只依赖浮窗
6. 运行 `npm run validate:sdd -- --app {app-code} --gate G3`

### step4-freeze 完成标准

| # | 条件 |
|---|------|
| F1 | `19-G2-A` 人工通过 |
| F2 | `03-PRD` 标 **冻结版** |
| F3 | `页面-路由-接口-数据表映射表.md` 完成 |
| F4 | `validate:sdd --gate G3` pass |

## 模板章节清单（必须与模板一致）

禁止沿用旧版「三章功能说明」骨架。权威章节：

| 章 | 标题 |
|----|------|
| — | 文档信息、版本变更记录、变更标记规范 |
| 一 | 项目概述 |
| 二 | 系统流程（含 2.4 评审会前检查清单） |
| **三** | **用户故事**（3.0～3.4） |
| **四** | **功能说明**（4.0 模块清单 + 4.x 六段式 + 4.2 检查） |
| 五 | 其他非功能需求 |
| 六 | 物理模型 |
| 七 | 标准编码 |
| 八 | 开发实施交付附录（8.1～8.3） |
| 附件 | 设计评审问题清单、评审结论 |

完整占位与表格结构 **以模板文件为准**，本 Skill 不重复粘贴全文。

## 禁止（Agent 红线）

- 步骤 2 用本 Skill 代替 SDD 模板 07/11
- 无 G2-A 人工通过即标 PRD「冻结版」
- 在 PRD 中字段级重复定义 API（须引用 SDD）
- `standard` 流程写 MVP 裁剪或分期不上线
- 仅有 PRD 文档而无全量可交互界面（步骤 3 越权进步骤 4）
- 将映射表仅写在 PRD §8.1 而不更新独立映射表文件

## 输出摘要（每步结束）

```markdown
## PRD 步骤摘要
- **模式**：step3-draft | step4-freeze
- **路径**：…/03-PRD设计评审文档.md
- **版本 / 状态**：Vx.y · 草案 | 冻结版
- **模块数 / US 数**：…
- **与 04/路由一致**：✅ / ❌
- **映射表**：已更新独立文件 ✅ / ❌
- **脚本**：validate:sdd --gate G3 — exit {code}
- **结论**：pass / fail / blocked
```
