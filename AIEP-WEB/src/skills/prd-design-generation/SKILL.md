# PRD 设计生成

## 目的

把需求基线转换为可执行 PRD，确保跨角色理解一致。PRD 关键章节未完成前，不允许进入 UI 或编码阶段。

## 触发场景

以下情况使用本技能：
- `requirements-clarification` 已完成
- 团队需要正式产品规格文档
- 需要产设研测统一对齐
- 需要支撑 UI-first 评审（先界面确认，再技术补齐）

## 必要输入

- `requirements-clarification` 输出的需求基线
- 业务约束（周期、资源、合规）

## 模板调用要求（强制）

1. 优先从 `{TEMPLATE_REPO_PATH}` 复制模板后填写（默认：`核心文档/AI+产品落地/02-子应用通用模板/`）。  
2. 本阶段必用模板：`03-PRD设计评审文档模板.md`。  
3. 如涉及界面方案补充，可同步参考 `02-产品设计文档模板.md`。  
4. 若模板路径不存在，必须自动生成同结构骨架，不得中断执行。  
5. 未按模板章节完整输出，阶段门禁判定为 `fail` 或 `blocked`（缺关键输入/依赖时）。

## 必交付物

1. 项目背景与问题定义
2. 目标与非目标
3. 用户角色与核心场景
4. 功能需求（FR）与非功能需求（NFR）
5. 端到端用户流程
6. 数据与 API 约束
7. 里程碑、发布策略、KPI 指标
8. 风险与回退方案
9. G2-A（前端交互样机评审）输入清单
10. G2-B（API/数据库/后端补齐评审）输入清单

## 执行步骤

1. 校验需求基线完整性
2. 将用户故事映射到 FR/NFR
3. 定义主流程与替代流程
4. 为每个目标补齐可度量 KPI
5. 输出分期计划（MVP / v1 / v2）
6. 复核是否与范围边界冲突

## 阶段门禁

满足以下条件才可进入 UI 阶段：
- 每条功能需求至少映射一个验收标准
- KPI 可量化且可追踪
- MVP 与后续范围清晰分离
- G2-A 与 G2-B 所需输入项已在 PRD 中明确标注

未通过则继续修订 PRD；若缺外部依赖或前置决策，状态标记为 `blocked`。
