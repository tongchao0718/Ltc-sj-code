# 需求澄清

## 目的

在进入 PRD 或编码前统一需求口径。未完成本技能产物，不允许进入下一阶段。

## 触发场景

以下情况必须使用本技能：
- 用户提出新功能或优化需求
- 需求存在歧义、冲突或约束缺失
- 团队即将进入 PRD 设计阶段

## 模板调用要求（强制）

1. 优先从 `{TEMPLATE_REPO_PATH}` 复制模板后填写（默认：`核心文档/AI+产品落地/02-子应用通用模板/`）。  
2. 本阶段必用模板：`01-需求说明书模板.md`。  
3. 若模板路径不存在，必须自动生成同结构骨架，不得中断执行。  
4. 若未按模板章节输出或缺失必填章节，阶段门禁判定为 `fail`。  
5. 输出结果需回填到子应用目录，不得直接改写模板母版文件。

## 必交付物

必须一次性输出以下内容：
1. **第1章 范围**：子应用编码、主路由前缀、适用版本、业务目标摘要
2. **第2章 功能需求**：需求清单 + 2.1 需求详尽化说明（六要素）
3. **第3章 非功能需求**：性能、安全、可用性、可维护性等可验收条目
4. **第4章 用户故事地图（概要）**：Story + 优先级 + 发布切片 + 验收摘要
5. **第5章 需求追踪矩阵**：需求到 PRD/测试/证据的映射
6. **第6章 变更记录**：版本、日期、摘要、责任人
7. **补充附录**：风险量化表、待确认项闭环表（可置于文末附录）

## 执行步骤

1. 用一段话复述本次目标。
2. 先做对话式澄清：至少提出 5 个关键问题消除歧义。  
3. 提取用户显式需求并建立“需求追溯表”。  
4. 推导隐性约束（时间、数据、平台、合规、存量系统影响）。  
5. 产出用户故事并标注优先级（P0/P1/P2）。  
6. 由用户故事生成验收标准（先功能，后非功能）。  
7. 补充边界场景与失败场景，并识别外部依赖。  
8. 输出风险量化表与待确认项闭环信息。  
9. 执行可测试性自检，未通过则继续澄清。  
10. 按固定顺序输出交付物。

## 对话澄清最小问题集（强制）

至少覆盖以下问题：
1. 本次需求要解决的核心痛点是什么？不解决会怎样？
2. 哪些用户角色会使用该功能？触发场景是什么？
3. 成功时系统输出什么？失败时如何提示或回退？
4. 会影响哪些现有功能？哪些行为必须保持不变？
5. 是否存在法律、平台、数据合规或审计约束？

## 阶段门禁

满足以下条件才可进入 PRD 阶段：
- 验收标准完整且可测试
- 范围边界无自相矛盾
- 待确认项已显式列出并给出闭环责任
- 用户故事已完成优先级划分
- 非功能验收标准已覆盖最小集合（性能/安全/可用性）
- 输出章节与 `01-需求说明书模板.md` 结构一致（1~6章）

若任一条件不满足，停留在本阶段继续澄清。

## 可测试性检查表（强制）

1. 每条验收标准是否包含可观察输入/输出或可度量阈值？
2. 是否包含测试条件（数据、触发动作、预期结果）？
3. 是否存在模糊词（如“高效”“友好”“大致”）？若有必须替换为可测阈值。
4. 每条验收标准是否有唯一编号并可追溯到用户故事？
