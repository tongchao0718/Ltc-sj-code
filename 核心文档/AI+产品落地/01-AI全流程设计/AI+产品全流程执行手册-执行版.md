# AI+产品全流程执行手册-执行版

### 执行版生成规则（强制）

1. 本文档是治理版的派生文档。  
2. 仅包含治理版中 `LEVEL=CORE` 规则。  
3. 本文档禁止直接手工编辑。  
4. 发生 `LEVEL=CORE` 规则变更时，必须重新生成本文档。  
5. 未重新生成状态标记 `blocked`，不得放行。

### 0. 启动页（一页可跑）

1. 目标  
   用最短路径把需求变成可发布版本。  
   主线固定：`需求生成 -> 需求确认 -> 界面生成 -> 界面确认 -> 开发测试 -> 发布上线`。

2. 角色  
   - 产品负责人（PO）  
   - 技术负责人（TL）  
   - 开发测试执行人（DEV/QA）

3. 唯一事实源（强制）  
   - API、数据模型、验收标准：只写在 `SDD`。  
   - 页面与交互：只写在 `PRD`。  
   - 其他文档禁止重复描述，只允许引用。

4. 三条红线（全流程通用）  
   - 红线1：`SDD` 未签署，禁止开发。  
   - 红线2：`PRD` 未冻结，禁止联调和提测。  
   - 红线3：测试未通过、CI/CD未通过、追溯链不完整，禁止发布。

---

### 1. 步骤1：需求生成

1. 输入  
   - 业务目标  
   - 问题描述  
   - 成功指标

2. 动作  
   - 明确范围：做什么、不做什么。  
   - 定义验收场景：正常、异常、边界。  
   - 标记流程类型：标准流程或 Fast Track。

3. 输出（唯一核心产物）  
   - `需求草案`

4. 放行条件（唯一）  
   - `需求草案` 完整，且 PO 确认进入评审。

---

### 2. 步骤2：需求确认

1. 输入  
   - `需求草案`

2. 动作  
   - 评审需求范围、优先级、验收标准。  
   - 完成 SDD 与技术补齐评审。  
   - 冻结需求边界与不做项。

3. 输出（唯一核心产物）  
   - `需求确认单`（引用：需求卡 + SDD + 技术补齐结论）

4. 放行条件（唯一）  
   - `需求确认单` 完整，且 PO 与 TL 双签。

---

### 3. 步骤3：界面生成

1. 输入  
   - `需求确认单`  
   - 页面范围清单

2. 动作  
   - 生成页面与交互方案初稿。  
   - 对齐 SDD 字段与状态定义。  
   - 补齐关键页面状态（loading/empty/error）。

3. 输出（唯一核心产物）  
   - `PRD 草案`

4. 放行条件（唯一）  
   - `PRD 草案` 覆盖全部需求范围与关键交互。

---

### 4. 步骤4：界面确认

1. 输入  
   - `PRD 草案`  
   - `需求确认单`

2. 动作  
   - 按 SDD 核对页面字段、状态、交互路径。  
   - 冻结页面清单和关键交互。  
   - 建立界面元素到 SDD 字段映射。

3. 输出（唯一核心产物）  
   - `PRD 冻结版`

4. 放行条件（唯一）  
   - 页面清单完成字段映射，PO 确认冻结。

---

### 5. 步骤5：开发测试

1. 输入  
   - `需求确认单`  
   - `PRD 冻结版`

2. 动作  
   - 按 SDD 实现，禁止口头改契约。  
   - 按 SDD 验收标准执行测试。  
   - 缺陷回写到任务与 SDD 映射记录。

3. 输出（唯一核心产物）  
   - `测试验收报告`

4. 放行条件（唯一）  
   - 验收标准通过率 100%，阻断级缺陷为 0。

---

### 6. 步骤6：发布上线

1. 输入  
   - `测试验收报告`  
   - CI/CD 执行结果

2. 动作  
   - 执行发布检查：版本、回滚、变更说明、监控阈值。  
   - 上线并完成追溯归档。  
   - 记录复盘结论和改进项。

3. 输出（唯一核心产物）  
   - `发布记录单`

4. 放行条件（唯一）  
   - CI/CD 全绿、回滚可用、追溯链完整。

---

### 7. Fast Track（小需求快速通道）

1. 适用条件（同时满足）  
   - 影响页面不超过 2 个。  
   - 不新增核心数据表。  
   - 不改变外部 API 主契约。  
   - 开发测试总工期不超过 3 人日。

2. 简化步骤  
   - 需求生成  
   - 需求确认  
   - 界面生成（并入界面确认）  
   - 界面确认  
   - 开发测试  
   - 发布上线

3. 必留产物  
   - `需求卡`  
   - `SDD-Lite`  
   - `测试验收报告`  
   - `发布记录单`

4. 禁用场景  
   - 涉及权限、计费、合规、审计。  
   - 涉及跨系统事务一致性。  
   - 涉及核心链路性能改造。

---

### 8. AI 使用规则（执行口径）

1. 基本规则  
   - AI 为加速器，不是流程门禁。  
   - AI 不可用，流程继续推进。

2. 强制规则  
   - AI 输出必须经过 Gate 校验。  
   - Gate 不通过，直接人工修订。

3. 失败兜底  
   - 失败定义：超时、幻觉、结构不合规、关键字段缺失。  
   - 兜底动作：切换人工模板直填。  
   - 响应时限：30 分钟内切换完成。

4. AI 可用于  
   - 需求拆解初稿  
   - SDD 草案生成  
   - 用例初稿生成  
   - 发布说明初稿

5. AI 不可替代  
   - SDD 签署  
   - 界面冻结确认  
   - 发布放行决策

---

### 9. 每日执行检查（站会直接使用）

1. 当前处于哪一步。  
2. 当前唯一产物是否齐全。  
3. 当前放行条件是否满足。  
4. 是否触发三条红线。  
5. 是否进入或退出 Fast Track。  
6. 是否触发 AI 兜底。  
7. 当前步骤是否与六步主线一致。
