# AI+产品全流程执行手册-治理版

### 0. 文档架构与派生机制（强制）

1. 单一事实源  
   - 治理版是唯一规则源。  
   - 执行版是治理版的派生展示层。

2. 规则标签（每条必填）  
   - `[LEVEL: CORE]`：进入执行版。  
   - `[LEVEL: EXTENDED]`：仅保留治理版。  
   - `[LEVEL: AUDIT]`：仅保留治理版（审计/合规）。

3. 变更触发  
   - 任一规则变更必须标注 `LEVEL`。  
   - 任一规则变更必须标注“是否影响执行版”（是/否）。  
   - `LEVEL=CORE` 变更未同步执行版（或未重新生成），状态标记 `blocked`。

4. 编辑边界  
   - 执行版禁止直接手工修改。  
   - 执行版仅允许由治理版筛选 `LEVEL=CORE` 后生成。

5. 一致性校验  
   - 必须满足：`执行版内容 ⊆ 治理版 LEVEL=CORE 规则集合`。  
   - 校验不通过，发布前门禁不放行。

### A. 门禁规则（不能怎么做）

1. [LEVEL: CORE] Gate-1：需求 -> SDD  
   - 禁止：无 `需求卡` 直接编写 SDD。  
   - 必须：`需求卡` 已签署。

2. [LEVEL: CORE] Gate-2：SDD -> 开发  
   - 禁止：SDD 未签署进入编码。  
   - 必须：SDD 覆盖 API/数据模型/验收标准并双签。

3. [LEVEL: CORE] Gate-3：界面 -> 联调提测  
   - 禁止：PRD 未冻结进入联调。  
   - 必须：PRD 冻结且字段映射完成。

4. [LEVEL: CORE] Gate-4：测试 -> 发布  
   - 禁止：缺陷未清零或 CI 失败仍发布。  
   - 必须：测试通过、CI/CD 全绿、追溯链完整。

---

### B. SDD 规范（不能降级）

1. [LEVEL: CORE] 强制结构  
   - 模块边界  
   - API 契约（输入/输出/错误码）  
   - 数据模型（字段/约束/索引）  
   - 验收标准（Given/When/Then）  
   - 非功能要求（性能/安全/可观测）  
   - 风险与回滚策略

2. [LEVEL: EXTENDED] 强制质量  
   - 每条验收标准必须可测试。  
   - 每个 API 必须定义错误处理。  
   - 每个关键字段必须可追溯来源与去向。  
   - 每个需求点必须映射到代码和测试。

3. [LEVEL: CORE] 强制变更  
   - 变更先改 SDD，再改代码。  
   - 变更必须记录版本、责任人、时间、原因。  
   - 未更新 SDD 的实现变更视为违规。

4. [LEVEL: CORE] 最小章节清单（必须齐全）  
   - 文档元信息（`sdd_id`、版本、负责人、日期）。  
   - 业务范围（做什么/不做什么）。  
   - API 契约。  
   - 数据模型。  
   - 页面状态机。  
   - 异常处理策略。  
   - 验收标准。  
   - 风险与回滚触发条件。  
   - 外部依赖清单。

5. [LEVEL: EXTENDED] API 契约字段约束  
   - 每个 API 必须包含：`path`、`method`、`request`、`response`、`errors`。  
   - 涉及列表查询时必须定义分页字段。  
   - 涉及写操作时必须定义幂等策略。  
   - 涉及受限资源时必须定义鉴权约束。

6. [LEVEL: EXTENDED] 数据模型字段约束  
   - 每个模型必须包含字段字典（名称、类型、是否必填、默认值）。  
   - 主键、外键、唯一约束必须显式定义。  
   - 索引策略必须说明（查询索引、唯一索引、组合索引）。  
   - 枚举字段必须提供取值口径与业务含义。

7. [LEVEL: CORE] 验收与测试映射约束  
   - 每条验收标准必须使用 Given/When/Then 结构。  
   - 每条验收标准必须包含 `acceptance_id`。  
   - 每个 `acceptance_id` 必须映射至少一个测试用例ID。  
   - 未建立映射关系状态标记 `blocked`。

8. [LEVEL: CORE] 全链路追溯约束  
   - 必须可追溯：`需求ID -> SDD条目 -> 代码提交 -> 测试用例 -> 发布单`。  
   - 追溯链缺失任一节点，测试放行与发布放行均不通过。

9. [LEVEL: CORE] 版本与兼容约束  
   - `major`：结构性变更或不兼容变更。  
   - `minor`：新增能力且向后兼容。  
   - `patch`：修订与缺陷修复，不改变结构。  
   - 每次版本变更必须声明兼容策略与回滚策略。

10. [LEVEL: AUDIT] SDD 机读校验约束  
   - 必须保留机读版本（JSON 或等价结构化格式）。  
   - 机读字段最小集必须覆盖：`sdd_id`、`features`、`api_contracts`、`data_model`、`acceptance`、`risk_and_rollback`。  
   - 发布前必须完成一次结构化校验并留痕。  
   - 校验记录缺失视为审计不通过。

---

### C. CI/CD 约束（不能绕过）

1. [LEVEL: CORE] 分支与合并  
   - 只允许受保护分支进入发布。  
   - 合并前必须通过自动检查。  
   - 禁止直接向发布分支手工提交。

2. [LEVEL: CORE] 必过检查  
   - 编译与静态检查通过。  
   - 单元测试与集成测试通过。  
   - 关键路径冒烟测试通过。  
   - 制品回滚验证通过。

3. [LEVEL: CORE] 发布控制  
   - 每次发布必须绑定 `发布记录单`。  
   - 每次发布必须有回滚步骤与责任人。  
   - 监控告警阈值必须提前生效。

4. [LEVEL: AUDIT] 审计追溯  
   - 必须可从发布版本反查：需求卡 -> SDD -> 测试报告 -> 发布记录。  
   - 追溯链缺失任一环节，发布无效。

---

### D. 执行与治理关系

1. 执行版用于一线交付。  
2. 治理版用于管理、审计、复盘。  
3. 冲突时以治理版红线与门禁为准。  
4. 豁免必须书面审批并注明有效期。

---

### E. 六步流程与 Gate 映射（强制口径）

1. [LEVEL: CORE] 六步流程标准顺序  
   - 需求生成 -> 需求确认 -> 界面生成 -> 界面确认 -> 开发测试 -> 发布上线。

2. [LEVEL: CORE] Gate 映射关系  
   - 需求生成：Gate 前准备，不放行。  
   - 需求确认：对应 Gate-1（需求->SDD）。  
   - 界面生成：Gate 前准备，不放行。  
   - 界面确认：对应 Gate-3（界面->联调提测）。  
   - 开发测试：对应 Gate-2（SDD->开发）与测试放行前检查。  
   - 发布上线：对应 Gate-4（测试->发布）。

3. [LEVEL: CORE] 执行约束  
   - 执行版步骤名称与顺序必须与本映射一致。  
   - 任一步骤变更必须同步更新映射关系。  
   - 映射关系未更新状态标记 `blocked`。
