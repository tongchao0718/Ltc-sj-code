# AI+产品全流程执行手册（V3.7）

## 1. 目的与使用方式

本手册用于将 `AI+产品落地方案` 转换为可直接执行的操作标准。  
适用对象：产品经理、设计、开发、测试、项目经理、安全负责人。  
执行原则：先分级脱敏，后调用 AI；先人工确认，后进入下一关卡。

### 1.1 适用范围与边界（新增）

适用范围：
- 需求边界清晰、业务规则稳定、跨系统依赖较少的中小型功能；
- 子应用新增页面、列表查询、表单处理、状态流转、基础报表等场景。

不适用范围（需走增强治理流程）：
- 高风险核心交易链路；
- 强合规审批链路；
- 高复杂算法与跨域强依赖场景。

## 1.2 子应用（项目）创建前置要求（强制）

在进入全流程 AI 化执行前，必须先完成子应用（项目）创建与基础治理准备。

必备项：
1. 已创建子应用代码目录与基础路由入口（`<app-code>` 唯一且语义化）。  
2. 已创建子应用文档目录：`docs/子应用文档/<app-code>/`。  
3. 已复制并初始化通用模板（至少 01~10），完成基础信息替换。  
4. 已指定角色责任人（产品、研发、测试、项目管理、安全）。  
5. 已完成信息分级与外部 AI 使用审批策略确认。  
6. 已建立版本追踪标识（`需求ID`、`sdd_id`、分支命名规则、发布单号规则）。  

准入规则：
- 未满足以上任一项，不得进入 Discover（D-01）。  
- 子应用目录与文档目录不一致，视为流程阻塞项（blocked）。  

### 1.2.2 模板主从关系（强制）

- `01-AI全流程设计`：规范母版与治理规则维护目录。  
- `02-子应用通用模板`：项目执行唯一填报模板目录。  
- 项目执行中如需填写模板，必须从 `核心文档/AI+产品落地/02-子应用通用模板/` 复制使用。  
- 禁止在 `01-AI全流程设计` 目录直接创建项目实例文档。  

### 1.2.1 模板补齐策略（强制）

新建子应用时，模板执行采用“先建骨架、后按关卡补齐”的策略：

1. **创建阶段**：一次性创建模板文件骨架（01~13）并完成基础元信息替换（`app-code`、版本、负责人）。  
2. **执行阶段**：按流程关卡逐步补齐内容，不要求在创建当日一次性写完全部模板。  
3. **门禁原则**：进入下一关前，仅检查“该关卡最小必需模板”是否完整。  

关卡最小必需模板建议：
- Discover（G1前）：`01`、`02`  
- Define（G2前）：`03`、`07`、`11`、`08`、`09`、`13`  
- Develop/Validate（G3/G4前）：`04` + 测试与映射相关产物  
- Deliver（G5前）：操作说明、版本变更记录、发布复盘相关产物  

不允许行为：
- 以“模板未一次性全部填写”为理由阻塞已满足当前关卡要求的任务推进。  
- 跳过当前关卡必需模板直接进入下一关。  

### 1.2.3 模板完整性门禁（强制）

为避免“精简版文档”进入评审或开发，新增模板完整性门禁：

1. 子应用文档必须按 `02-子应用通用模板` 的章节结构输出，不得随意删减必填章节。  
2. G2-B 前必须完成模板一致性检查：需求、PRD、SDD 三份文档需满足模板同结构（允许内容为空占位，但章节不得缺失）。  
3. SDD JSON 必须满足模板约定字段（含 `roles`、`story_map`、`api_contracts` 字段级契约、`data_model`、`state_machine`、`exception_strategy`、`acceptance`、`risk_and_rollback`）。其中 `story_map` 必须包含 `acceptance_ref` 与 `acceptance_standard`。  
4. 若存在模板缺项，关卡状态标记 `blocked`，不得进入客户确认或开发阶段。  

### 1.2.4 文档详尽度门禁（强制）

为避免“章节存在但内容过浅”导致后续返工，新增详尽度门禁：

1. `需求说明书` 每条核心需求需至少说明：业务目标、输入来源、处理规则、输出结果、异常处理、验收标准。  
2. `PRD设计评审文档` 每个模块需至少说明：功能需求、流程、交互/权限、数据存储、接口映射、注意事项。  
3. G2-B 前必须通过“防遗漏检查”：需求-PRD-SDD 三文档在编号、范围、异常场景和验收引用上保持一致。  
4. 出现“仅一句话描述核心功能”或“缺少异常与边界定义”的文档，门禁状态必须标记 `blocked`。  
5. 评审结论必须给出“详尽度结论”（通过/有条件通过/不通过），未给结论不得放行。  
6. `PRD设计评审文档` 第三章“功能说明”中，每个模块的“（1）功能需求”至少包含 3 条能力点描述，且至少覆盖“对象范围 + 处理规则 + 结果约束”三类信息；不满足则视为不通过。  
7. 所有涉及“用户故事地图”的文档必须填写“验收标准（摘要）”，并与验收脚本（AT）保持一致；未填写或不一致视为 `blocked`。  
8. `详细设计文档` 禁止一句话描述模块实现；每个模块必须具备“输入、处理、输出、异常与恢复、验收”五类信息，否则 G3 不放行。  
9. `数据库设计文档` 必须覆盖全部物理模型，并提供每个模型的建表 SQL（DDL）示例；缺失任一模型或SQL时，G3/G4 标记 `blocked`。  
10. `数据库设计文档` 的“物理模型清单（全量）”必须按“每个字段一行”输出（禁止“一张表一行”汇总写法）；且主键命名禁止通用 `id`，必须为“`表名_id`”（如 `protocol_template_id`），并与 SQL DDL 完全一致；不一致时门禁标记 `blocked`。  
11. `数据库设计文档` 必须新增“标准编码清单”章节；且“物理模型清单（全量）”必须增加“表中文名、字段中文名”两列，“标准编码名（系统功能）”仅在字段实际使用标准编码时填写，未使用必须留空；违反任一项时 G3/G4 标记 `blocked`。  
12. 编码实现必须遵循 `核心文档/框架核心文档/创建指南.md` 与 `核心文档/框架核心文档/子应用打包指南.md`：子应用目录结构、入口文件、路由模式、打包脚本与产物结构不符合时，G3/G4 标记 `blocked`。

## 2. 总体流程与关卡

流程顺序：`需求澄清 -> 需求冻结 -> 设计评审（前端交互样机） -> 客户确认（界面效果） -> 技术补齐评审（API/数据库/后端） -> 开发实现 -> 测试验收 -> 发布上线 -> 复盘优化`

必过关卡：
1. G1 需求冻结
2. G2-A 前端交互样机评审与界面效果确认
3. G2-B 技术补齐评审（API/数据库/后端）
4. G3 开发完成（DoD）
5. G4 测试放行
6. G5 发布与复盘归档

| 关卡 | 准入（进入前） | 准出（放行前） | 不通过回退步骤 | 责任人 |
|------|------|------|------|------|
| G1 需求冻结 | 需求草案、范围初稿、风险初稿 | 冻结需求、不做项、验收标准签字 | 回退 D-03/D-04 | 产品经理 |

| G2-B 技术补齐评审 | G2-A确认版界面、字段映射草案、接口草案 | API契约冻结、数据库模型确认、后端实现方案确认 | 回退 F-06/F-07 | 产品经理/开发负责人 |

| G5 发布归档 | 发布清单、回滚方案、监控预案 | 发布说明、复盘报告、归档完成 | 回退 R-01/R-04/R-06 | 项目经理 |

## 2.2 全流程 AI 化标准（强制）

为确保“全流程 AI 化且不缺环节”，每个阶段必须同时满足“标准输入 + 标准输出 + 关卡门禁”。

| 阶段 | 标准输入（最小集） | 标准输出（最小集） | 门禁要求 |
|------|------|------|------|
| Discover | 业务目标、痛点、约束、脱敏输入 | `需求说明书`、需求评审问题清单 | G1 冻结签字（建议 T+0） |
| Define | 冻结需求、原型草案、接口草案 | 前端交互样机确认记录、客户确认版 `SDD`、`PRD设计评审文档`、映射表 | G2-A 界面确认 + G2-B 技术补齐评审通过（建议 T+1~T+2） |
| Develop | 确认版 SDD、任务拆解、联调计划 | 可运行代码、覆盖矩阵、自测记录 | G3 DoD + 代码评审 + SDD映射通过（按迭代计划） |
| Validate | 联调版本、测试计划、用例集 | 测试报告、缺陷闭环清单、UAT结论 | G4 P0/P1 清零（按提测窗口） |
| Deliver | 发布清单、回滚方案、监控预案 | 发布说明、复盘报告、行动清单 | G5 归档完成（发布后 T+1 完成复盘草稿） |

补充规则：
1. 任一阶段标准输出缺失，禁止进入下一关。  
2. AI 输出仅为草稿，必须通过人工复核与门禁校验。  
3. 需求变更必须先更新 SDD，再更新代码与测试资产。  

## 2.3 Skills 使用规范（强制）

为统一 AI 行为与交付标准，新增 skill 驱动执行规则：

1. 全流程必须先调用 `ai-dev-stage-gate` 判断当前阶段与缺失输入。  
2. 每个阶段只允许调用对应阶段 skill：
   - Discover/需求澄清：`requirements-clarification`
   - Define/PRD设计：`prd-design-generation`
   - G2-A 界面样机与效果确认：`ui-generation`
   - G2-B 技术补齐评审（API/数据库/后端）：`code-development`（前置评审模式）
   - Develop/编码实现：`code-development`
   - Validate/测试放行：`test-validation`
3. 未通过当前阶段 skill 的 Stage Gate，不得进入下一阶段。  
4. 阶段切换时，必须再次调用 `ai-dev-stage-gate` 输出 `pass/fail` 结论。  
5. skill 输出仅作为标准草稿，关键结论仍需人工确认签字。  
6. 所有 skill 的检索入口统一为：`AIEP-WEB/src/skills/SKILLS-DIRECTORY.md`。
7. 发生文档规则、模板或 skill 变更时，必须执行 `核心文档/框架核心文档/文档联动更新规范.md`，未联动完成状态标记 `blocked`。

### 2.3.1 UI-first 执行规则（强制）

为提升确认效率，Define 阶段采用“先界面、后技术补齐”：

1. G2-A 允许先实现前端交互样机（可使用 Mock 数据）。  
2. G2-A 通过后，必须进入 G2-B 补齐 API、数据库、后端实现方案。  
3. G2-B 未通过，不得进入 G3 开发与提测。  
4. 若 G2-A 确认后发生界面字段调整，必须同步回写 API 与库表映射。  
5. 所有“界面确认”与“技术确认”结论需留痕归档。

### 2.3.2 Skills 门禁执行模板（可复制）

```text
当前任务：<任务名>
当前阶段：<discover/define/develop/validate/deliver>

步骤1：调用 ai-dev-stage-gate，确认当前阶段缺失项
步骤2：调用对应阶段 skill，产出必交付物
步骤3：再次调用 ai-dev-stage-gate，给出门禁结论（pass/fail）
步骤4：若 fail，列出缺失项并补齐；若 pass，进入下一阶段
```

### 2.3.3 专项 Skill 触发矩阵（可选增强）

在不改变五阶段主线门禁的前提下，可按场景触发专项 skill：

| 触发场景 | 触发阶段 | 建议 Skill 组合 | 执行约束 |
|------|------|------|------|
| 新页面设计或界面改版 | Define / UI | `enhance-prompt` -> `stitch-design` | 仅用于界面生成，不替代 PRD 与验收门禁 |
| 需要统一设计语言 | Define / UI | `design-md`（必要时 `taste-design`） | 必须先有需求边界，避免先画后补需求 |
| 设计稿转前端组件 | Develop | `react:components`（必要时 `shadcn-ui`） | 代码仍需通过 `code-development` 的追溯与验证 |
| 汇报演示与 walkthrough | Deliver | `remotion` | 仅作为交付展示增强，不作为放行依据 |
| 连续页面自动迭代 | UI / Develop | `stitch-loop` | 标记为实验能力，需负责人审批后启用 |

补充规则：
1. 专项 skill 不得替代主线 skill。  
2. 专项 skill 触发后，仍必须回到 `ai-dev-stage-gate` 执行 pass/fail 判断。  
3. 专项 skill 输出若与主线产物冲突，以主线门禁产物为准并回退修订。  
4. 统一检索入口：`AIEP-WEB/src/skills/SKILLS-DIRECTORY.md`。

## 2.2.1 需求更新执行要求（强制）

当子应用存在新增需求或需求变更时，必须执行“先文档、后实现、再交付”的闭环流程：

1. **变更登记与分级**：在 `01-需求说明书` 中登记变更类型（新增/调整/删除）、影响范围（模块/接口/库表/测试/发布）与优先级。  
2. **需求与设计同步更新**：至少同步更新 `01-需求说明书`、`03-PRD设计评审文档`、`05-SDD-vx.y.md`、`06-SDD-vx.y.json`，并保持编号与映射一致。  
3. **研发与测试资产联动**：涉及实现时，必须同步更新详细设计、数据库设计、验收脚本（Given/When/Then）与映射表。  
4. **版本与留痕强制**：每次需求更新必须记录版本变更（含变更摘要、责任人、日期、影响项），不得只改正文不记变更。  
5. **门禁阻断规则**：若需求、PRD、SDD（md/json）任一未更新或不一致，G2-B/G3 状态必须标记为 `blocked`，禁止提测与发布。  
6. **详尽度规则**：更新后的需求与 PRD 必须补齐输入、处理、输出、异常、验收五类信息，禁止“仅改标题不补内容”。  
7. **PRD功能需求规则**：若 `PRD` 第三章任一模块“功能需求”不足 3 条或无法支撑研发拆解，必须回退补齐后再评审。  
8. **环节联动规则**：上一环节文档一旦更新，必须同步更新下一环节文档后方可推进。示例：需求说明书更新后，必须同步更新 PRD；PRD 更新后，必须同步更新 SDD（md/json）；SDD 更新后，必须同步更新详细设计/测试资产。未联动更新时，关卡状态标记为 `blocked`。  
9. **代码变更反向联动规则**：若已发生功能代码（含功能界面、交互逻辑、接口调用、状态流转）调整，文档更新必须以“人工明确提出更新要求”为前置条件；未收到明确要求时，不默认触发文档改写。  
10. **编码后文档更新方式（强制）**：编码完成后禁止直接依赖自动生成或自动覆盖文档；仅在产品/研发负责人提出明确更新要求后，方可执行文档更新并复核关键差异（接口、状态机、数据模型、验收项）。被要求更新但未完成回写与复核时，G3/G4 状态标记为 `blocked`。  
11. **轻量化执行规则（强制）**：进入 G3 后，文档仅保留“执行必需最小集”（任务拆解、联调/提测准入、风险阻塞）；禁止无明确价值的扩写。若新增文档或章节，必须说明执行目的与使用时机。  

最小更新清单（每次需求更新至少覆盖）：

## 3. 通用执行动作（所有环节都要做）

## 3.1 通用步骤

1. 信息分级（L1-L4）。
2. 脱敏处理（如需）。
3. 填写 `AI 输入审批单`（L2+ 外部 AI 必填）。
4. 生成 AI 输出草稿。
5. 人工复核与修订。
6. 版本归档与留痕。

## 3.2 通用方法

- 方法 M1：双通道校验（AI 草稿 + 人工业务复核）。  
- 方法 M2：反向验证（用测试视角反推需求是否可测）。  
- 方法 M3：追溯校验（需求条目能否映射到页面、接口、测试用例）。  
- 方法 M4：最小发布单元（优先上线可独立验证的最小功能集合）。

### 3.6.1 参数字典（通用）

## 3.7 SDD直连编码模式（强制）

### 3.7.1 目标与规则

为支持 AI 化转型，新增“SDD 直连编码”执行模式：  
**简单需求 -> 结构化 SDD -> 客户确认 -> 直接编码**。

强制规则：
1. 客户确认版 `SDD` 为编码唯一事实源。  
2. `PRD`、原型、会议纪要作为输入参考，不得覆盖已确认 SDD。  
3. 发生需求变更时，必须先升级 SDD 版本，再允许代码变更。  
4. 未完成 SDD 确认记录，不得进入 G3 开发关卡。
5. SDD 必须双格式归档：`SDD-vx.y.md`（可读） + `SDD-vx.y.json`（机读）。

### 3.7.2 SDD最小必填章节

1. 文档元信息（`sdd_id/version/owner/date`）  
2. 业务范围（`in_scope/out_of_scope`）  
3. 用户角色与权限矩阵  
4. 功能点清单（含优先级）  
5. API 契约（请求/响应/错误码）  
6. 数据模型（字段、类型、约束）  
7. 页面交互与状态机（`loading/success/empty/error`）  
8. 异常处理策略（超时、断网、4xx、5xx）  
9. 验收脚本（Given/When/Then）  
10. 风险与回滚策略

### 3.7.3 SDD JSON Schema（V1.0）

```json
{
  "sdd_id": "SDD-20260411-001",
  "version": "1.0.0",
  "owner": "产品经理",
  "stage": "define",
  "feature_name": "示例功能",
  "in_scope": ["功能A", "功能B"],
  "out_of_scope": ["不做项X"],
  "roles": [
    { "role": "管理员", "permissions": ["read", "write", "export"] }
  ],
  "features": [
    { "id": "F-01", "name": "查询列表", "priority": "P0", "acceptance_ref": ["AT-01"] }
  ],
  "api_contracts": [
    {
      "id": "API-01",
      "path": "/api/example/list",
      "method": "GET",
      "request": { "keyword": "string", "pageNo": "number", "pageSize": "number" },
      "response": { "code": "number", "message": "string", "data": "object", "requestId": "string" },
      "errors": ["400", "401", "500"]
    }
  ],
  "state_machine": [
    { "page": "ListPage", "states": ["loading", "success", "empty", "error"] }
  ],
  "exception_strategy": [
    { "type": "timeout", "action": "show_retry" },
    { "type": "network_offline", "action": "show_offline_hint" }
  ],
  "acceptance": [
    {
      "id": "AT-01",
      "given": "用户在列表页",
      "when": "切换筛选条件",
      "then": "触发真实请求并重置到第一页"
    }
  ],
  "rollback_plan": "发布后异常可在30分钟内回滚到上一个稳定版本"
}
```

### 3.7.4 SDD 校验规则（新增）

1. 必须包含：`features/api_contracts/state_machine/acceptance`。  
2. `features` 至少 1 条且 `id` 唯一。  
3. `api_contracts` 至少 1 条，且必须含 `path/method/request/response`。  
4. `acceptance` 至少 1 条 Given/When/Then。  
5. 出现占位符 `<...>`、`TODO`、空对象 `{}` 视为不通过。  
6. 校验失败必须回退到 F-02/F-03 修订，不得进入 G2-B 放行。

## 3.8 自动化门禁与转型度量（强制）

### 3.8.1 双门禁机制（人工+自动）

- **G2-A（界面确认）自动门禁**：
  - 前端交互样机评审通过；
  - 客户界面效果确认记录已归档；
  - 样机问题清单闭环。
- **G2-B（技术补齐）自动门禁**：
  - SDD JSON Schema 校验通过；
  - 页面-路由-接口-数据表映射完整；
  - 客户确认记录已归档。
- **G3（开发完成）自动门禁**：
  - 代码提交必须关联 `sdd_id`；
  - 需求到代码覆盖矩阵通过；
  - Given/When/Then 验收脚本全部可执行。

### 3.8.2 自动校验失败处理

1. 任一自动校验失败，关卡状态标记为 `blocked`。  
2. 责任人在 24 小时内修复并重新触发校验。  
3. 连续两次失败，升级到项目经理与模块负责人复审。  
4. 超过 SLA 未闭环，纳入当期复盘与绩效改进项。

### 3.8.3 AI转型核心 KPI（新增）

1. `SDD首轮通过率` = 首轮通过的 SDD / SDD 总数。  
2. `确认后变更率` = 客户确认后变更的需求条目 / 总需求条目。  
3. `SDD-代码一致率` = 可映射到 SDD 的实现项 / 总实现项。  
4. `AI采纳率` = AI 产出被采纳的任务项 / AI 产出总任务项。  
5. `需求到可编码时长`（Lead Time）：需求输入到 SDD 确认完成时长。  
6. `上线后回滚率`：上线后触发回滚版本 / 发布总版本。

### 3.8.4 KPI目标建议（首期）

- SDD 首轮通过率：>= 70%  
- 确认后变更率：<= 15%  
- SDD-代码一致率：>= 95%  
- AI 采纳率：>= 60%  
- Lead Time：较现状缩短 >= 30%

### 3.8.5 自动门禁脚本契约（新增）

为确保 G2-A/G2-B/G3 自动门禁可落地，脚本执行必须遵循以下输入输出约定：

1. **执行入口**：在 PR 检查流水线和发布前流水线均执行。  
2. **输入文件**：
   - `SDD-vx.y.json`
   - `映射表.md`（页面-路由-接口-数据表）
   - `门禁检查配置.json`（可选）
3. **输出结果（JSON）**：必须包含 `gate`、`status`、`failed_items`、`checked_at`。
4. **状态枚举**：`passed`、`failed`、`blocked`。  
5. **失败处理**：状态为 `failed/blocked` 时，流水线必须阻断并生成修复建议。

示例输出：

```json
{
  "gate": "G2-B",
  "status": "failed",
  "failed_items": [
    "SDD missing api_contracts",
    "mapping table not found"
  ],
  "checked_at": "2026-04-28T16:50:00+08:00"
}
```

### 3.8.6 SDD规范驱动的轻敏捷执行（强制）

为避免流程过重，采用“轻敏捷 + 强门禁”模式，不强制引入固定会议编排，仅保留最小执行动作：

1. 任务进入开发前，必须完成 `SDD-vx.y.md/json` 基线确认，并绑定 `sdd_id`。  
2. 任务拆解必须对齐 `SDD.features`，每项任务至少绑定一个 `feature_id` 与 `acceptance_ref`。  
3. 开发过程中若实现偏离 SDD，必须先完成 SDD 变更并通过门禁，再继续实现。  
4. 提测前必须输出“SDD映射核对结果”：已覆盖条目、未覆盖条目、风险条目。  
5. 迭代结束时仅要求沉淀三项指标：`SDD首轮通过率`、`SDD-代码一致率`、`确认后变更率`，不强制指定会议形式。

### 3.8.7 SDD规范驱动的 CI/CD 融合（强制）

为将 SDD 驱动落到流水线，CI/CD 必须设置“提交前、合并前、发布前”三级检查：

1. **提交前（Pre-commit/本地）**：  
   - 校验 `SDD json schema`、`feature_id` 命名、`acceptance_ref` 引用完整性。  
   - 失败处理：本地阻断提交，给出缺失字段与修复建议。
2. **合并前（PR CI）**：  
   - 校验 `需求/PRD/SDD/md-json/映射表` 是否联动更新；  
   - 校验代码变更是否可映射到 `sdd_id + feature_id`（提交信息、PR 模板或变更清单）；  
   - 校验关键验收脚本是否已更新并可执行。  
   - 任一失败，PR 状态为 `blocked`，禁止合并。
3. **发布前（CD Gate）**：  
   - 校验 G2-B/G3/G4 门禁结果均为 `passed`；  
   - 校验当前发布包对应的 `sdd_id` 与发布单一致；  
   - 校验回滚方案已关联当前 SDD 版本。  
   - 任一失败，禁止发布并自动通知项目经理与发布负责人。

配套规则：
1. PR 模板必须包含：`sdd_id`、`feature_id`、`acceptance_ref`、风险与回滚说明。  
2. 分支命名建议：`feature/<sdd_id>-<feature_id>-<topic>`。  
3. 发布说明必须增加“SDD 映射摘要”章节：本次上线覆盖的 SDD 条目、未覆盖条目、风险条目。

## 3.9 工具链落地建议（新增）

1. 需求输入层：统一“需求输入卡”（Markdown + JSON）。  
2. SDD 生成层：固定 Prompt 模板 + SDD Schema 校验器。

5. 追溯治理层：`需求ID -> sdd_id -> 分支 -> 发布单号` 全链路映射。

## 9. 异常分支处理（防遗漏）

## 9.1 常见异常与应对

| 场景 | 触发条件 | 应对动作 | 升级路径 |
|------|------|------|------|
| 需求反复变更 | 设计后新增核心诉求 | 启动变更评审，重新冻结范围 | 产品经理 -> 项目经理 |
| 客户确认冲突 | 客户意见与技术约束冲突 | 提供 A/B 方案与影响评估 | 产品经理 -> 技术负责人 |
| AI 输出质量差 | 结果不完整或错误 | 补充上下文，改用校验 Prompt 复核 | 执行人 -> 负责人 |
| 涉密风险 | 发现未脱敏信息输入 | 立即止损，封禁会话，24h复盘 | 执行人 -> 安全负责人 |
| 上线后高优缺陷 | P0/P1 线上暴露 | 启动应急发布或回滚 | 项目经理 -> 发布负责人 |

## 9.2 异常处理提示词

```text
请基于以下异常情况，输出“应急处置方案”：
1) 现象与影响
2) 根因假设
3) 立即止损动作（T+0）
4) 24小时内修复计划
5) 防复发机制
```

## 9.3 异常分级与响应 SLA

| 级别 | 触发示例 | 响应时限 | 处置要求 | 升级路径 |
|------|------|------|------|------|
| P0 | 涉密泄露、核心功能不可用 | 15 分钟内 | 立即止损、必要时回滚、启动应急群 | 执行人 -> 安全负责人 -> 项目经理 |
| P1 | 主流程受阻但可降级 | 1 小时内 | 发布修复计划，24 小时内修复 | 执行人 -> 模块负责人 -> 项目经理 |
| P2 | 非主流程问题 | 1 个工作日内 | 纳入迭代修复清单 | 执行人 -> 模块负责人 |
| P3 | 文档类或体验优化项 | 3 个工作日内 | 评估后排期 | 执行人 -> 产品经理 |

## 9.4 判级口径与值班机制

### 9.4.1 判级口径

1. 影响范围 > 50% 用户或核心主流程中断：判定 P0。  
2. 主流程可降级但无法正常闭环：判定 P1。  
3. 非主流程缺陷，不影响核心交付：判定 P2。  
4. 文档或体验优化建议：判定 P3。  

### 9.4.2 值班机制

- 工作时段：由模块负责人一线响应，项目经理二线协调。  
- 非工作时段：值班工程师一线响应，安全负责人/发布负责人二线值守。  
- 值班表更新频率：每周更新一次并归档。  
- 交接要求：值班交接必须包含“未闭环问题清单 + 当前风险等级”。

## 10.1 命名与归档规范（强制）

### 命名规范

`<项目>-<模块>-<文档类型>-<版本>-<YYYYMMDD>.md`

示例：`AIEP-DEV-sample-app-PRD-V1.2-20260411.md`

### 归档目录

1. 子应用规范根目录：`docs/子应用文档/<app-code>/`
2. 需求与设计：`docs/子应用文档/<app-code>/01-需求与设计/`
3. 研发与测试：`docs/子应用文档/<app-code>/02-研发与测试/`
4. 发布与复盘：`docs/子应用文档/<app-code>/03-发布与复盘/`
5. AI 治理留痕：`docs/子应用文档/<app-code>/04-AI治理与审计/`

### 子应用归档强制规则

1. 所有 AI 生成文档必须按子应用独立存放，禁止跨应用混放。  
2. 文档命名必须包含 `<app-code>` 与版本号，确保可检索与可追溯。  
3. 跨子应用公共规则仅放在 `AI全流程设计/`，具体执行记录必须回收到各 `<app-code>` 目录。  
4. 未按子应用归档视为 G5 不通过。  

### 版本策略

- 主版本（V1.0 -> V2.0）：流程或模板结构性变更。  
- 次版本（V1.1 -> V1.2）：规则补强、章节新增。  
- 修订版本（V1.2.1）：错字、表述修正，不改变流程。
