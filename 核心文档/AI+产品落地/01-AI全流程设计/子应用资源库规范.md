# 子应用资源库规范（借鉴 Axhub Make `_resources` 模式）

> **文档状态**：V0.1 草案  
> **日期**：2026-06-08  
> **定位**：步骤 3「界面生成」的可复用资源真值层；与 SDD/PRD 双真值并存，**不替代** Gate 与契约文档。  
> **关联 Skill**：`sub-app-resources`、`design-review-pre-g2a`、`project-memory`（`.cursor/skills/`）

---

## 一、为什么需要 `_resources`

| 问题（现状） | `_resources` 解法 |
|--------------|-------------------|
| 每次生成页面都从框架规范文档「重新解读」 | 子应用级 **theme / 组件 / Mock 数据** 一次沉淀、持续复用 |
| 多页风格漂移（色板、间距、组件选型不一） | `theme.json` + `components-manifest.json` 作为步骤 3 硬引用 |
| Agent 跨会话重复问已确认决策 | `00-项目记忆.md` 记录决策与否决项 |
| G2-A 前缺少结构化预审 | `design-review-pre-g2a` 对照资源库 + PRD 草案出报告 |

**原则**：`_resources` 是 **体验层辅助真值**；契约仍以 SDD 为准，评审主文档仍以 `03-PRD` 为准。

## 二、目录结构（Web 子应用）

```text
AIEP-WEB/src/apps/{app-code}/
├── _resources/                    # 本规范核心（步骤 3 初始化，步骤 4 前须与代码一致）
│   ├── README.md                  # 资源库说明、来源、更新记录
│   ├── theme.json                 # 色板 / 字号 / 间距 / 圆角 token
│   ├── components-manifest.json   # 本应用可用组件清单与用途
│   └── mock-data/                 # 演示数据（按页面或实体分文件）
│       ├── README.md
│       └── {entity}.json          # 如 customers.json、orders.json
├── views/ …
├── components/ …
└── mock/ …                        # 运行时 Mock（可引用 _resources/mock-data）
```

**文档侧配套**（跨会话记忆，不在代码目录）：

```text
AIEP-WEB/src/docs/子应用文档/{app-code}/
└── 04-AI治理与审计/
    └── 00-项目记忆.md             # 由 project-memory Skill 维护
```

## 三、文件职责

### 3.1 `theme.json`

从 `platform_type` 对应设计规范 Skill 提取的最小 token 集，供步骤 3 批量出页时 **强制引用**。

| 字段 | 说明 |
|------|------|
| `version` | 语义化版本，变更须写 `README.md` 更新记录 |
| `platform_type` | 与 `01-需求说明书.md` 一致 |
| `source` | 权威规范文档路径（如 `营销设计规范.md`） |
| `colors` | 主色、功能色、中性色 |
| `spacing` | 页面级 / 区块级间距（如 YXUI 24/16） |
| `typography` | 字号阶梯、行高 |
| `radius` | 圆角 token |

**禁止**：在 `theme.json` 中定义 API 字段或业务规则（归属 SDD）。

### 3.2 `components-manifest.json`

记录本应用 **允许使用** 的组件及其场景，减少步骤 3 随意造轮子。

```json
{
  "version": "1.0",
  "platform_type": "web-marketing",
  "shared_from": "marketing-demo/components",
  "entries": [
    { "name": "MdFilterForm", "use_for": ["列表查询区"], "import": "../components/MdFilterForm.vue" }
  ]
}
```

### 3.3 `mock-data/`

- 步骤 3 演示用数据，字段名 **须与 SDD 草案或 SDD-Lite 一致**  
- 步骤 5 替换真实 API 时，以 SDD 为准校对，可保留 `_resources/mock-data` 作单测/fixture 参考

### 3.4 `00-项目记忆.md`

| 板块 | 内容 |
|------|------|
| 已确认决策 | PO/客户已口头或书面确认、尚未写入 PRD 的要点 |
| 明确否决 | 客户否决过的方案（Agent 不得重复提议） |
| 风格偏好 | 与本应用相关的非常规偏好 |
| 待办 | 跨会话未完成的动作（≤5 条） |
| 变更日志 | 日期 + 摘要 |

**维护时机**：`ai-full-process-design` 每次会话启动读取；步骤 3/4 有决策变更时由 `project-memory` 更新。

## 四、与六步主线的挂接

| 步骤 | `_resources` 要求 |
|------|-------------------|
| 1～2 | 无要求 |
| **3 入口** | 调用 `sub-app-resources` 初始化或校验 `_resources/` |
| **3 进行中** | 新页面须引用 `theme.json` + `components-manifest.json`；Mock 优先来自 `mock-data/` |
| **3→4 切换前** | 调用 `design-review-pre-g2a`，报告须 pass 或仅有「可接受警告」 |
| **4** | 客户确认后，PRD 冻结须与资源库 + 页面标注一致 |
| 5～6 | `_resources` 可演进，但不得与已冻结 PRD/SDD 冲突（须先走变更流程） |

### 4.1 与「原型即 PRD」的统一出口

步骤 3 完成页面后，推荐流程：

1. `prd-page-annotation` 挂载角标浮窗  
2. 角标编号 ↔ `01-需求说明书` SA-xx 一致  
3. `03-PRD` 由标注 **反向同步** 模块章节（非双轨各写）  
4. G2-A 走查以 **可跑界面 + 标注** 为主视图，`03-PRD` 为归档真值

## 六、验收检查（步骤 3→4 补充项 R1～R5）

在 U1～U6 之外，步骤 3 进入 4 前须满足：

| # | 检查项 |
|---|--------|
| R1 | `_resources/theme.json` 存在且 `platform_type` 与需求说明书一致 |
| R2 | `_resources/components-manifest.json` 覆盖 P0 页面用到的组件 |
| R3 | P0 页面 Mock 数据可追溯到 `_resources/mock-data/` 或 `mock/` |
| R4 | `design-review-pre-g2a` 报告已落盘且无 blocking 项 |
| R5 | `00-项目记忆.md` 已更新至当前会话（无 stale 待办） |

报告落盘路径：`04-AI治理与审计/G2-A预审报告.md`
