本子应用支持两种模式（**必须明确当前处于哪种，并在 §7/§8 按模式交付**）：

| 表名 | 索引名 | 类型 | 字段 | 用途 |
|------|--------|------|------|------|

### 4.2 数据一致性约束

- 状态字段统一使用 §3 标准编码，禁止魔法字符串散落。
- 关键写操作保留 `request_id` / `trace_id` 便于全链路追踪。
- 异步任务、同步任务须持久化状态与最近执行时间。
- 外键删除策略：`RESTRICT`（默认）或 `CASCADE` 须在 §7 逐表注明。
- 权限与敏感字段（账号、证件号）遵循最小权限与脱敏展示规则。

## 8. SQL 清单（全量 DDL，强制）

```sql
-- 方言：MySQL 8.x
-- Schema：<schema_name>

CREATE TABLE protocol_template (
  protocol_template_id BIGINT NOT NULL AUTO_INCREMENT COMMENT '协议模板主键ID',
  template_code        VARCHAR(64)  NOT NULL COMMENT '模板编码',
  template_name        VARCHAR(128) NOT NULL COMMENT '模板名称',
  status               VARCHAR(20)  NOT NULL DEFAULT 'draft' COMMENT '状态，标准编码 TASK_STATUS',
  created_at           DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at           DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (protocol_template_id),
  UNIQUE KEY uk_protocol_template_code (template_code),
  KEY idx_protocol_template_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='协议模板主表';
```

> **要求**  
> 1. 每张表一段可执行 DDL，字段 COMMENT 与 §7「字段中文名」一致。  
> 2. §7 字段数量、类型、主键、必填、索引与 §8 **一一对应**。  
> 3. 外键、CHECK（若用）写在 DDL 或 §4 中并交叉引用。
