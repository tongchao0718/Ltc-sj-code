## 0. 文档真值层级（强制）

## 1. 适用范围

本模板用于多子应用项目的标准文档交付，覆盖需求、产品、评审、设计、数据库、操作及 AI 全流程治理模板。

适用对象：

- 新建子应用立项与评审
- 既有子应用迭代版本评审
- 研发、测试、产品跨角色协同

## 3. 使用步骤

1. 在 `docs/子应用文档/<app-code>/` 下**先创建四段目录**（与 `sample-app` 一致）：
   - `01-需求与设计/`、`02-研发与测试/`、`03-发布与复盘/`、`04-AI治理与审计/`
2. 从 `02-子应用通用模板/` 复制模板到**对应阶段目录**（禁止在 `<app-code>/` 根目录平铺）。
3. 将文件名中的“模板”改为正式文档名。
4. 全局替换占位符：
   - `<子应用名称>`
   - `<app-code>`
   - `<主路由前缀>`
   - `<版本号>`
   - `<评审轮次>`
   - `<负责人>`
5. 按真实业务补齐接口、库表、标准编码、测试证据。
6. 评审前使用 PRD 内“评审会前检查清单”逐项勾选。
7. AI 驱动项目需同步使用 07~14 模板，形成“SDD -> JSON -> 校验 -> 门禁 -> 配置 -> 任务拆解 -> 版本治理”闭环。

### 3.2 机读校验命令（仓库根目录）

```bash
npm run validate:sdd -- --app <app-code> --gate G2
npm run coverage:acceptance -- --app <app-code>
```

说明见 `AIEP-WEB/scripts/README.md`；PR 模板见 `.github/pull_request_template.md`。

## 3.1 关键模板填写口径（新增）

- `02-产品设计文档模板.md`：定位为“产品边界说明”，建议轻量维护（定位、信息架构、交互边界、风格、发布策略），不重复承载 PRD 的接口/数据细节。
- `04-详细设计文档模板.md`：强制补齐每模块“输入、处理、输出、异常与恢复、验收”；并补充“关键页面时序”“标准编码覆盖说明”“设计与实现边界”。
- `05-数据库设计文档模板.md`：`2.库表概览` 与 `4.核心字段说明` 为可选摘要；**强制**以 `7.物理模型清单（全量，字段级）` + `8.SQL清单（全量DDL）` + `9.对齐检查清单` 为交付基准；须含文档元信息、表关系、索引设计、Demo/落地双模式说明。
- `05-数据库设计文档模板.md`：物理模型清单必须包含“表中文名、字段中文名”；“标准编码名（系统功能）”按需填写（字段用到才填，未用留空）。
- `07-SDD模板-可机读版.md`：建议补齐“技术与工程约束、统一返回结构、数据库命名规则、业务异常定义、测试生成规则”，与 G2/G3 门禁保持一致。
- `14-G3开发任务拆解表模板.md`：G3 前建议强制填写，确保 `SA-F-API-DB-AT` 覆盖可追溯、任务可排期、DoD 可核验。

### 模板补齐策略（强制）

- 新建子应用时，先创建模板骨架（01~14）并完成基础信息替换。
- 模板内容按关卡逐步补齐，不要求创建当日一次性写完全部模板。
- 进入下一关前，仅检查当前关卡最小必需模板是否完整。
- 不得跳过当前关卡必需模板直接推进流程。

## 4. 维护规则

- 文档采用“单文档主干 + 版本变更记录 + 局部变更标记”模式。
- PRD 作为评审主文档，需求、产品、详细设计、数据库文档与其保持对齐。
- 模板更新后，必须同步检查并更新对应执行手册门禁条款，避免“模板已改、门禁未改”导致执行歧义。
- G3 阶段遵循轻量化执行：仅维护任务拆解、联调/提测准入、风险阻塞三类执行文档；无人工明确要求时不新增扩展文档。
- 编码与打包必须遵循 `核心文档/框架核心文档/创建指南.md` 与 `核心文档/框架核心文档/子应用打包指南.md`（目录结构、入口、路由、打包配置、产物结构）。
- 每次评审会后至少更新：
  - PRD 版本变更记录
  - 评审问题清单状态
  - 需求追踪矩阵
  - 评审结论
- 所有生成文档按子应用独立存放，禁止跨子应用混放。

## 5. 建议交付门槛

- 必填项完整率 >= 90%
- 关键接口契约冻结完成
- 关键异常链路可验证
- DoD 可执行且有证据产物路径
