# AI工作流程方案（SDD直连编码）

## 一、目标与适用范围

### 1.1 目标

在团队 AI 转型阶段，建立一条标准化流程：  
**简单需求 -> AI 生成可机读 SDD -> 客户确认 -> AI/研发直接编码 -> 测试验收 -> 发布上线**。

### 1.2 适用范围

- 需求边界清晰、业务规则稳定、跨系统依赖较少的中小型功能；
- 子应用新增页面、列表查询、表单处理、状态流转、基础报表等场景；
- 不适用于高风险核心交易、复杂算法、强合规审批链路（需走增强流程）。

## 二、核心原则

- **单一事实源**：客户确认后的 SDD 是编码唯一依据。
- **结构化优先**：SDD 必须机读化、字段化，避免自然语言歧义。
- **确认后开工**：未完成客户确认的 SDD 不得进入编码阶段。
- **门禁前移**：在 SDD 阶段就固化接口、状态、异常、验收标准。
- **全程可追溯**：需求版本、SDD版本、代码版本、上线版本一一对应。

## 三、端到端流程

## 3.1 流程总览

1. 需求输入与澄清  
2. AI 生成 SDD（结构化）  
3. 人工复核（产品/架构/测试）  
4. 客户评审与确认  
5. AI 辅助编码实施  
6. 自动化测试与回归  
7. 验收发布与复盘

## 3.2 各阶段详细说明

### 阶段A：需求输入与澄清（T+0）

输入：
- 简单需求描述（业务目标、用户角色、触发场景）
- 约束条件（时间、合规、性能、系统边界）

动作：
- 产品经理使用“需求收敛模板”补全必填字段；
- 补齐“不可做范围（Out of Scope）”。

输出：
- `需求输入卡`（可机读 JSON/YAML）

准入门槛：
- 目标、范围、角色、约束、验收口径至少 80% 明确。

### 阶段B：AI 生成 SDD（T+0 ~ T+1）

动作：
- 将需求输入卡喂给 AI，生成 SDD 初稿；
- SDD 必须包含：模块、流程、接口、数据模型、状态机、异常策略、验收用例。

输出：
- `SDD-v0.1.md` + `SDD-v0.1.json`（推荐双格式）

准出标准：
- 字段完整；
- 无“待补充”关键项；
- 关键交互均有请求与回写定义。

### 阶段C：内部复核（T+1）

参与角色：
- PM（业务正确性）
- 架构/研发（可实现性）
- 测试（可测性）

动作：
- 使用“SDD复核清单”逐条评估；
- 输出问题单并回改。

输出：
- `SDD-v1.0`（内部通过版）

门禁：
- 未通过复核不得发客户确认。

### 阶段D：客户确认（T+1 ~ T+2）

动作：
- 客户确认功能范围、关键流程、边界规则、验收标准；
- 变更项回写到 SDD 并升级版本。

输出：
- `SDD-v1.x-客户确认版`
- 客户确认记录（邮件/会议纪要/审批截图）

门禁：
- 无确认记录不得编码。

### 阶段E：AI辅助编码（T+2 ~ T+N）

动作：
- 基于 SDD 自动生成任务切片（前端、后端、测试）；
- AI 按切片生成代码骨架、接口实现、测试用例；
- 人工进行代码评审与安全检查。

输出：
- 可运行代码；
- 单元测试、集成测试脚本；
- 变更说明与迁移说明（如有）。

门禁：
- 代码实现与 SDD 字段逐项映射，不得“自由发挥”偏离确认范围。

### 阶段F：测试验收与发布（T+N）

动作：
- 按 SDD 的 Given/When/Then 执行功能验收；
- 覆盖异常场景：超时、断网、4xx/5xx、部分失败；
- 发布前核对版本映射关系。

输出：
- 验收报告；
- 发布记录；
- 复盘结论。

## 四、SDD标准结构（AI可识别）

建议 SDD 同时输出 Markdown（便于阅读）+ JSON（便于 AI/工具消费）。

### 4.1 必填章节

1. 文档元信息（版本、作者、日期、关联需求ID）
2. 业务目标与范围（In Scope / Out of Scope）
3. 用户角色与权限矩阵
4. 业务流程（主流程、备选流程、异常流程）
5. 功能需求明细（按功能点编号）
6. 数据模型与字段定义
7. API契约（请求、响应、错误码）
8. 页面与交互状态机
9. 非功能要求（性能、安全、审计、可观测）
10. 验收标准（Given/When/Then）
11. 风险与回滚策略

### 4.2 示例字段（JSON）

```json
{
  "sdd_id": "SDD-2026-001",
  "version": "1.0.0",
  "feature_name": "客户查询增强",
  "scope": {
    "in": ["多条件查询", "分页联动", "导出任务"],
    "out": ["客户画像算法重构"]
  },
  "api_contracts": [
    {
      "path": "/api/customer/list",
      "method": "GET",
      "request": {
        "keyword": "string",
        "status": "string",
        "pageNo": "number",
        "pageSize": "number"
      },
      "response": {
        "code": "number",
        "message": "string",
        "data": "object",
        "requestId": "string"
      }
    }
  ],
  "acceptance": [
    {
      "given": "用户在客户列表页",
      "when": "切换状态筛选条件",
      "then": "触发真实请求并重置到第一页"
    }
  ]
}
```

## 五、角色分工

- **产品经理（Owner）**：需求收敛、SDD生成与修订、客户确认组织。
- **AI工程支持/研发**：提示词优化、代码生成策略、实现落地。
- **架构师**：把关接口一致性、可扩展性、安全性。
- **测试工程师**：基于 SDD 生成并执行测试用例。
- **客户代表**：确认范围、规则和验收口径。

## 六、门禁机制（必须执行）

### 6.1 准入规则

- 需求未结构化不得生成 SDD；
- SDD 未通过内部复核不得对客确认；
- SDD 未确认不得进入编码。

### 6.2 准出规则

- 代码与 SDD 一致性 >= 95%；
- 核心验收用例通过率 = 100%；
- 异常场景覆盖率 = 100%；
- 发布前文档、代码、测试版本一致。

## 七、工具链建议

- 需求输入：标准化模板（Markdown + JSON）
- SDD生成：AI 助手 + 固定 Prompt 模板
- 编码执行：AI 代码助手 + Git 流程
- 验收追踪：测试平台或检查单模板
- 版本追踪：需求ID、SDD ID、分支名、发布单号打通

## 八、度量指标（转型成效）

- SDD 首轮通过率
- 从需求到可编码周期（Lead Time）
- 需求变更率与返工率
- 缺陷密度（提测前/上线后）
- 客户确认后变更次数

## 九、风险与应对

- **风险：AI误解业务语义**  
  应对：增强需求输入模板，增加术语字典与反例。

- **风险：SDD过于抽象，无法直接编码**  
  应对：强制补齐 API、字段、状态机、验收脚本四大块。

- **风险：客户确认后仍频繁变更**  
  应对：设置变更窗口和变更分级，超阈值触发二次评审。

- **风险：团队依赖 AI 但缺少审查**  
  应对：保留人工复核门禁，关键链路必须人工签字。

## 十、落地节奏（首期6周）

- 第1周：模板定版（需求输入卡、SDD模板、复核清单）
- 第2周：选 1 个简单需求试点，跑通全流程
- 第3-4周：扩展到 2-3 个子应用功能
- 第5周：沉淀标准 Prompt、最佳实践与反例库
- 第6周：发布团队标准作业程序（SOP）并全员执行

## 十一、执行红线

- 未客户确认的 SDD 禁止编码；
- 无验收脚本的需求禁止提测；
- 关键异常未覆盖的功能禁止上线；
- 任何口头需求变更必须回写 SDD 后再实施。
