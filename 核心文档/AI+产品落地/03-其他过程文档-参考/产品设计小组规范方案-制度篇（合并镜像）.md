## 0. 总则

### 0.1 文档性质

| 类别 | 本方案承担的内容 |
|------|------------------|
| **制度** | 谁必须做什么、禁止做什么；台账、评审、对接的刚性要求 |
| **规范** | 对接方式、会议与纪要、归档与版本、对内对外通道 |
| **标准** | 准入/准出门槛、最小交付物、记录字段、AI 分级与短长周期口径 |

### 0.2 适用范围与对象

- **必须**适用：产品设计小组在编成员；在项目中承担 PO/PM 职责的人员，执行本方案中与角色匹配的条款。  
- **适用范围**：客户与项目组对接、六步主线上的设计侧活动、设计类评审与交付物、AI 辅助设计的声明与留痕、小组评审与问题台账。

### 0.3 条款等级（全文统一）

| 用语 | 执行含义 |
|------|----------|
| **必须** | 不满足则**不得**进入下一环节，或不得视为已合规完成 |
| **禁止** | 违规行为**必须**纠正；与治理红线重叠时按《治理版》处理 |
| **应当** | 默认遵守；确需偏离须 **PO 书面同意**并注明有效期与原因 |
| **建议** | 质量与效率导向；部门可发文升格为应当/必须 |

### 0.4 效力与变更

- **冲突处理**：与《治理版》《执行版》条款冲突时，以**《治理版》**为准。  
- **变更**：由文档维护人维护版本；重大变更同步知会小组并在《执行手册》主入口可检索处注明（若部门有统一变更记录模板，从其规定）。

## 1. 目标与原则标准

### 1.1 战略目标（建议与部门年度/OKR 对齐）

- 工具驱动的 AI 设计工作流，设计产物可驱动开发与 AI 生成代码。  
- 对接前置：需求共创与体验前置，控制偏差与返工。  
- 业务专项深耕，形成可识别领域能力。  
- AI 双轨：短期提效与规范沉淀，中长期产品内 AI 能力。

### 1.2 执行原则（必须 / 禁止）

- **必须**：API、数据模型、验收标准只写在 **SDD**；页面与交互只写在 **PRD**；对客户与项目组的**范围与验收类承诺**落在**确认单或纪要**，并引用 SDD/PRD 条目或版本。  
- **必须**：需求与接口**先结构化**（需求卡、SDD 草案），再展开原型与视觉大工作量。  
- **必须**：AI 使用按第 5 章 **L1～L4** 声明阶次递进，**禁止**跳过本方案规定的人工 Gate（含签署、冻结、机读校验留痕等与治理版一致的门禁）。  
- **必须**：评审发现的规范类问题进入**问题台账**，含责任人、整改时限与关闭证据。  
- **禁止**：以即时通讯聊天记录作为**唯一**契约依据（可与 3.6 并用）。

## 2. 组织与职责制度

### 2.1 角色职责标准

| 角色 | 必须履行的职责 |
|------|----------------|
| 产品负责人（PO） | 范围与优先级裁定；需求确认与界面冻结类签署；客户对齐总责 |
| 技术负责人（TL） | SDD 可实现性与技术风险；需求确认单双签之一；配合联调提测门禁 |
| 产品经理（PM） | 需求卡、SDD/PRD 编制与版本维护；共创与评审组织；追溯字段维护 |
| 测试代表（QA） | 验收可测性；用例与 `acceptance_id` 映射（与治理版一致） |
| 业务方向负责人（部门任命时） | 该方向模板、指标与规则资产维护；该方向项目设计口径抽检 |

### 2.2 业务方向负责人制度

- **必须**：部门划定的每个**核心方向**有且仅有 **1 名**负责人（或书面委托的代行机制）。  
- **应当**：项目承接与人力投入向核心方向倾斜；非核心方向投入**应当**有 PO 备案说明。

## 3. 对接制度与客户 / 项目规范

### 3.1 需求共创（制度）

- **禁止**：在未就业务目标、核心场景、验收口径形成**书面确认载体**前，以大规模原型或开发排期对外承诺。  
- **必须**：对齐载体以**原型级或关键流程 DEMO**为主；场景描述优先于纯功能清单堆砌。

### 3.2 与客户对接规范

| 类别 | 标准 |
|------|------|
| 会议与纪要 | **必须**产出纪要：决议、待决项、行动项（责任人+日期）；对客户承诺以纪要或确认单为准 |
| 资料与保密 | **必须**在对外发送含真实或敏感数据的原型/样例前完成**脱敏与审批**（遵循部门脱敏检查单或等效流程） |
| 分歧与升级 | **应当**先书面列差异再上会；**超过 2 个工作日**未闭合的分歧升级至 PO + 客户侧负责人 |
| 联系人变更 | **必须**书面确认并更新干系人台账（见 3.6） |

### 3.3 双周客户共创（规范）

- **必须**：默认**双周**至少一次共创（项目加密须 PO 书面确认频次）；**会前 24h** 发出议程与上轮行动项完成情况。  
- **必须**：会议产出**书面结论**（同意 / 修改点 / 待决项）。  
- **应当**：会前备齐需求卡摘要、原型链接或录屏、SDD 开放问题清单（若有）。

### 3.4 三阶段确认（标准）

| 阶段 | 确认内容 | 责任方 | 必须产出的证据 |
|------|----------|--------|----------------|
| 需求阶段 | 业务目标、范围、成功指标、不做项 | PO + 客户 | 需求确认单（引用需求卡 + SDD 摘要） |
| 原型阶段 | 核心体验、关键状态与异常、与 SDD 字段一致 | PO/PM + 客户 | PRD 冻结前评审纪要 + 字段映射表 |
| 上线前 | 预期效果、灰度、回滚与监控 | PO + TL + 客户 | 上线确认记录（可与发布检查合并） |

### 3.5 快速验证（规范）

- **建议**：**1～2 周**一轮可见成果（原型或 DEMO）。  
- **必须**：快速验证**不替代**《执行版》Gate；结论**必须**回写需求卡或 SDD 后再进入签署与开发排期。

### 3.6 对外承诺与变更追溯（制度）

- **必须**：对外范围与验收承诺对应**需求 ID 与 SDD 版本号**。  
- **禁止**：口头变更未在下次共创或书面流程中**追认**即纳入开发排期。

### 3.7 项目组对内对接（制度）

| 事项 | 标准 |
|------|------|
| 干系人台账 | **必须**每项目维护：PO、TL、PM、客户对接人、研发接口人、测试接口人；变更 **24h 内**更新 |
| 对接通道 | **必须**：契约类信息以需求卡、SDD、确认单、纪要为准；即时群**禁止**作为唯一契约载体 |
| 内部对齐 | **应当**与项目组周会/迭代会同步：当前六步位置、唯一产物是否齐、是否触红线；阻塞升级 PO+TL |
| 排期衔接 | **禁止**在未满足「需求确认 / SDD 签署 / PRD 冻结」对应门槛时由产品侧**单方承诺**开发完成日期 |
| 资料权限 | **必须**按项目台账开通工具与仓库权限；涉密链接**应当**标注密级 |

## 4. 流程与门禁执行标准

### 4.1 标准主线（与《执行版》一致，不得改序）

`需求生成 → 需求确认 → 界面生成 → 界面确认 → 开发测试 → 发布上线`

### 4.2 门禁执行标准（举证要求）

| 环节 | 必须满足的举证 |
|------|----------------|
| 需求确认 | 需求卡齐备；SDD 技术补齐评审完成；**需求确认单** PO+TL **双签** |
| 开发准入 | **SDD 已签署**（同治理红线） |
| 联调 / 提测 | **PRD 已冻结**；**界面元素 ↔ SDD 字段映射表**完成 |
| 发布 | 测试、CI/CD、追溯链符合《治理版》Gate-4 |

### 4.3 Fast Track

- **必须**：适用条件与必留产物**完全**遵循《执行版》Fast Track；**禁止**省略 SDD-Lite 与测试验收报告。

### 4.4 工具链标准（规范）

- **应当**：原型/交互在部门选定工具产出（**图颜**等），链接写入 PRD。  
- **应当**：需求卡、确认单、纪要在部门选定文档库归档（**九功**等），可追溯版本。  
- **必须**：SDD 使用部门模板（含机读字段）；AI 产出仅为初稿，**未经人审签署不得**作为开发唯一依据。

## 5. AI 使用制度与分级标准（L1～L4）

### 5.1 制度性要求

- **必须**：项目或需求维度**声明当前 AI 阶次**（L1～L4 之一）；越阶使用须 PO **书面同意**并说明风险承担。  
- **必须**：AI 输出经 **Gate 校验**（结构、事实、与 SDD/模板一致性）；不通过则人工修订或降级为 L1。  
- **必须**：触发《执行版》兜底规则时，**30 分钟内**切换人工模板直填并完成留痕说明。

### 5.2 分级标准表

| 阶次 | 名称 | 允许范围 | 必须的人工 Gate |
|------|------|------------|-------------------|
| L1 | 单点辅助 | 润色、纪要草稿、清单草稿等局部文本 | 事实与术语校对（建议 5～15 分钟量级） |
| L2 | 单环节结构化 | 固定模板内**单类**结构化块（如仅 API 表、仅 GWT 初稿） | 模板缺项检查；PO 或业务方向负责人抽检 |
| L3 | 多环节链式 | 同一需求内链式多产物；机读片段传递 | 全量 SDD 复核清单；`acceptance_id` 与用例映射 |
| L4 | 流程嵌入 + 质量门禁 | L3 + 机读校验、追溯字段、模板/组件库提示 | 机读校验**留痕**；发布前追溯链完整 |

### 5.3 递进标准（建议，可部门升格为应当）

- 新项目或新成员：**建议**先 **L1 满 2 个迭代**且无严重「AI 结构类」问题再默认 **L2**。  
- **建议**：连续多个需求 SDD 一次复核通过后，项目可升级到 **L3**。  
- **必须**：**L4** 仅在已具备机读校验能力与模板库基线的子应用中启用（启用须 PO 确认）。

### 5.4 六步与推荐阶次（标准）

| 六步 | 推荐阶次 | 禁止 |
|------|----------|------|
| 需求生成 | L1～L2 | AI 单独签署需求卡定稿 |
| 需求确认 | L2～L3 | AI 替代 SDD **签署** |
| 界面生成 | L1～L2 | — |
| 界面确认 | L1 | AI 替代 PRD **冻结**决策 |
| 开发测试 | L2～L3 | AI 替代测试放行 |
| 发布上线 | L1 | AI 替代发布放行决策 |

## 6. 模板与资产标准

- **必须**：SDD 机读最小字段集与章节要求**不得**低于《治理版》B 章及 `07-SDD模板-可机读版.md`。  
- **应当**：数据模型命名、枚举口径与部门数据模型模板一致。  
- **应当**：PRD 引用可复用组件时注明**组件 ID**（行业/部门组件库有则从其编号规则）。  
- **建议**：经评审的 AI 提示片段纳入小组片段库，**禁止**个人私存与口径冲突的「地下提示词」作为官方依据。

## 7. 评审制度与问题台账标准

### 7.1 小组定期评审（制度）

- **必须**：按部门发文周期开展小组设计评审（频次不得低于**每季度一次**，除非部门另有规定）。  
- **必须**：评审覆盖四维度：**设计规范性、业务合理性、数据模型完整性、工具与 AI 应用熟练度**。  
- **应当**：采用成果展示 + 交叉点评 + 领导指导；优秀案例纳入知识库索引。

### 7.2 问题台账（标准字段）

- **必须**字段：问题描述、类型（命名/链路/工具/AI 等）、责任人、整改时限、验证方式、关闭日期、证据链接。  
- **必须**闭环：**评审 → 反馈 → 整改 → 复盘**；重复问题**应当**转化为模板或检查项更新。

### 7.3 设计类评审准入与交付物标准

#### 7.3.1 评审类型与准入

| 评审类型 | 会前必须齐 | 必须形成的结论 |
|----------|------------|----------------|
| 需求与范围评审 | 需求卡草案、目标与不做什么 | 是否进入需求确认；开放问题清单 |
| SDD 技术补齐评审 | SDD 含治理版最小章节 | 补齐项；是否可进入双签 |
| SDD 签署评审 | SDD 完整版、追溯字段初填 | PO+TL 双签 |
| PRD/原型（对内） | PRD 草案、原型链接、与 SDD 差异说明 | 修改项与冻结条件 |
| PRD 冻结前（对客户） | PRD 草案、字段映射表 | 客户书面或纪要确认 |
| 上线前设计核对 | PRD 冻结版、SDD 当前版、变更说明 | 是否满足上线前确认 |

#### 7.3.2 六步最小交付物（设计侧）

| 六步 | 最小交付物 | 接收方 | 签署 / 准出 |
|------|------------|--------|-------------|
| 需求生成 | 需求草案/需求卡 | PO、项目组 | PO 确认进入评审 |
| 需求确认 | 需求确认单 | TL、研发 | PO+TL 双签 |
| 界面生成 | PRD 草案、原型链接 | PO、客户预览 | — |
| 界面确认 | PRD 冻结版、字段映射表 | 研发、测试 | PO 冻结确认 |
| 开发测试 | 契约变更说明（若有） | 研发、测试 | **必须先改 SDD** |
| 发布上线 | 产品可见变更摘要 | 运维/业务 | 发布记录单依统一流程 |

- **必须**：交付物在**项目约定文档库**归档；文件名或元数据含**版本、日期、sdd_id/需求 ID**。  
- **禁止**：仅以个人目录或离线文件作为**唯一**正式版本。  
- **必须**：签署后、冻结后版本**锁定版本号**，变更按《治理版》变更条款执行。

## 8. AI 应用清单与短 / 长期制度

### 8.1 短 / 长期分界（标准，起止由部门发文锁定）

| 维度 | 短期 | 长期 |
|------|------|------|
| 目标 | 设计人时下降、一次通过率提升、L 阶次稳定 | 若干 **AI 驱动核心业务功能**可持续运营 |
| 范围 | 工作流内 L1～L4、模板、机读校验、清单量化 | 产品内嵌入流程的智能能力 |
| 立项验收 | 每季度 1～2 提效场景；季度末对照指标复盘 | 业务+技术+安全合规；达「可运营」算落地 |
| 风险 | Gate 与兜底；**禁止**未签 SDD 大规模 AI 写码 | **禁止**因产品 AI 弱化 SDD/追溯红线 |

### 8.2 清单记录标准

- **短期清单每条必须含**：场景名称、当前 L 阶次、基线人时/次、目标人时/次、质量指标、责任人、本季度状态。  
- **长期清单每条必须含**：能力名称、业务痛点、嵌入流程、成功指标、数据与权限依赖、目标上线窗口、Owner。

### 8.3 试点与产品能力（规范）

- **应当**：每季度选择 1～2 场景做阶次加深试点并登记结果。  
- **建议**：中长期路线图中单列 AI 产品能力里程碑，与业务方向负责人机制挂钩。

## 9. 度量标准（KR，建议纳入部门统计）

| 维度 | 指标（建议） |
|------|----------------|
| 对接与需求 | 书面确认率；行动项按期关闭率；需求变更率（先建基线再谈降幅） |

| 专项能力 | 核心方向可复用资产数；核心方向项目占比 |

## 10. 落地执行检查单

> 供 PM **每迭代自查**、站会抽查、评审**会前准入**使用；勾选即表示已举证（链接或附件）。

### 10.1 对接与干系人

- [ ] 干系人台账为当前版本（含客户对接人）  
- [ ] 本轮共创/确认会纪要已归档，行动项有责任人日期  
- [ ] 无「仅存在于群聊」的契约性承诺  

### 10.2 流程与门禁证据
