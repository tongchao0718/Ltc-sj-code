# AI+产品全流程执行手册（V1.3）

## 1. 目的与使用方式

本手册用于将 `AI+产品落地方案` 转换为可直接执行的操作标准。  
适用对象：产品经理、设计、开发、测试、项目经理、安全负责人。  
执行原则：先分级脱敏，后调用 AI；先人工确认，后进入下一关卡。

---

## 2. 总体流程与关卡

流程顺序：`需求澄清 -> 需求冻结 -> 设计评审 -> 客户确认 -> 开发实现 -> 测试验收 -> 发布上线 -> 复盘优化`

必过关卡：
1. G1 需求冻结
2. G2 设计评审与客户确认
3. G3 开发完成（DoD）
4. G4 测试放行
5. G5 发布与复盘归档

## 2.1 关卡准入/准出操作表（强制）

| 关卡 | 准入（进入前） | 准出（放行前） | 不通过回退步骤 | 责任人 |
|------|------|------|------|------|
| G1 需求冻结 | 需求草案、范围初稿、风险初稿 | 冻结需求、不做项、验收标准签字 | 回退 D-03/D-04 | 产品经理 |
| G2 设计评审 | PRD草案、原型草案、接口草案 | 客户确认记录、技术可实现结论 | 回退 F-02/F-03/F-04 | 产品经理 |
| G3 开发完成 | 开发任务拆解、联调计划 | DoD通过、代码评审通过、自测通过 | 回退 V-03/V-05 | 开发负责人 |
| G4 测试放行 | 测试计划、用例集、联调版本 | P0/P1清零、UAT通过、回归通过 | 回退 T-03/T-04 | 测试负责人 |
| G5 发布归档 | 发布清单、回滚方案、监控预案 | 发布说明、复盘报告、归档完成 | 回退 R-01/R-04/R-06 | 项目经理 |

---

## 3. 通用执行动作（所有环节都要做）

## 3.1 通用步骤

1. 信息分级（L1-L4）。
2. 脱敏处理（如需）。
3. 填写 `AI 输入审批单`（L2+ 外部 AI 必填）。
4. 生成 AI 输出草稿。
5. 人工复核与修订。
6. 版本归档与留痕。

## 3.2 通用方法

- 方法 M1：双通道校验（AI 草稿 + 人工业务复核）。  
- 方法 M2：反向验证（用测试视角反推需求是否可测）。  
- 方法 M3：追溯校验（需求条目能否映射到页面、接口、测试用例）。  
- 方法 M4：最小发布单元（优先上线可独立验证的最小功能集合）。  

## 3.3 通用 AI 提示词模板（可复制）

```text
你是资深[角色]。请基于以下输入输出结构化结果。

【任务目标】
<写清本次要完成什么，不超过3条>

【输入材料】
<需求/原型/接口/问题清单等>

【约束条件】
1) 不得虚构未提供信息
2) 输出必须可执行、可验收
3) 必须标注风险与待确认项

【输出格式】
1. 结论摘要（3-5条）
2. 可执行步骤（编号）
3. 产出物清单
4. 风险与待确认项（含建议）
```

## 3.4 防泄密前置提示词（可复制）

```text
请执行“AI输入安全审查”，按以下格式输出：
1) 输入内容分级（L1-L4）及依据
2) 是否含敏感信息（账号、隐私、密钥、商业条款）
3) 脱敏建议（逐项替换）
4) 是否允许提交外部AI（是/否 + 理由）
5) 仍需人工审批项
```

## 3.5 结构化输出与重试策略（避免漏项）

### 3.5.1 标准输出 Schema（通用）

```json
{
  "summary": ["结论1", "结论2"],
  "steps": [
    { "id": "S1", "action": "动作", "owner": "角色", "output": "产物" }
  ],
  "risks": [
    { "level": "high|medium|low", "issue": "问题", "mitigation": "措施" }
  ],
  "pending_confirmations": ["待确认项1"]
}
```

### 3.5.2 重试策略

1. 首次输出不完整：补充上下文后重试一次。  
2. 二次仍不完整：切换“校验提示词”进行反向检查。  
3. 仍失败：人工接管，并记录为“AI 失败案例”纳入复盘。  

### 3.5.3 校验提示词（可复制）

```text
请审查以下输出是否满足：完整性、可执行性、可验收性、无敏感信息泄露。
按以下格式输出：
1) 缺失项
2) 歧义项
3) 风险项
4) 修订建议（可直接替换）
```

## 3.6 参数字典与 JSON 校验规则（统一口径）

### 3.6.1 参数字典（通用）

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `task_id` | string | 是 | 任务唯一ID | `D-20260411-001` |
| `stage` | enum | 是 | 阶段 | `discover/define/develop/validate/deliver` |
| `owner` | string | 是 | 责任角色 | `产品经理` |
| `priority` | enum | 否 | 优先级 | `P0/P1/P2/P3` |
| `due_date` | string | 否 | 截止日期（YYYY-MM-DD） | `2026-04-15` |
| `risk_level` | enum | 是 | 风险等级 | `high/medium/low` |

### 3.6.2 输出校验规则

1. 必须包含 `summary`、`steps`、`risks` 三个一级字段。  
2. `steps` 至少 3 个步骤，且每项必须含 `id/action/owner/output`。  
3. `risks` 至少 1 条，且风险等级必须在 `high/medium/low` 中。  
4. 不允许空字符串、占位符 `<...>` 作为最终交付内容。  
5. 校验失败时必须进入“3.5.2 重试策略”。

### 3.6.3 校验提示词（可复制）

```text
请对以下JSON输出执行结构校验：
1) 必填字段是否齐全
2) 枚举值是否有效
3) 是否存在占位符或空值
4) 是否满足可执行最小粒度（>=3步骤，>=1风险）
输出：通过/不通过 + 错误列表 + 修复建议
```

## 9. 异常分支处理（防遗漏）

## 9.1 常见异常与应对

| 场景 | 触发条件 | 应对动作 | 升级路径 |
|------|------|------|------|
| 需求反复变更 | 设计后新增核心诉求 | 启动变更评审，重新冻结范围 | 产品经理 -> 项目经理 |
| 客户确认冲突 | 客户意见与技术约束冲突 | 提供 A/B 方案与影响评估 | 产品经理 -> 技术负责人 |
| AI 输出质量差 | 结果不完整或错误 | 补充上下文，改用校验 Prompt 复核 | 执行人 -> 负责人 |
| 涉密风险 | 发现未脱敏信息输入 | 立即止损，封禁会话，24h复盘 | 执行人 -> 安全负责人 |
| 上线后高优缺陷 | P0/P1 线上暴露 | 启动应急发布或回滚 | 项目经理 -> 发布负责人 |

## 9.2 异常处理提示词

```text
请基于以下异常情况，输出“应急处置方案”：
1) 现象与影响
2) 根因假设
3) 立即止损动作（T+0）
4) 24小时内修复计划
5) 防复发机制
```

---

## 9.3 异常分级与响应 SLA

| 级别 | 触发示例 | 响应时限 | 处置要求 | 升级路径 |
|------|------|------|------|------|
| P0 | 涉密泄露、核心功能不可用 | 15 分钟内 | 立即止损、必要时回滚、启动应急群 | 执行人 -> 安全负责人 -> 项目经理 |
| P1 | 主流程受阻但可降级 | 1 小时内 | 发布修复计划，24 小时内修复 | 执行人 -> 模块负责人 -> 项目经理 |
| P2 | 非主流程问题 | 1 个工作日内 | 纳入迭代修复清单 | 执行人 -> 模块负责人 |
| P3 | 文档类或体验优化项 | 3 个工作日内 | 评估后排期 | 执行人 -> 产品经理 |

---

## 9.4 判级口径与值班机制

### 9.4.1 判级口径

1. 影响范围 > 50% 用户或核心主流程中断：判定 P0。  
2. 主流程可降级但无法正常闭环：判定 P1。  
3. 非主流程缺陷，不影响核心交付：判定 P2。  
4. 文档或体验优化建议：判定 P3。  

### 9.4.2 值班机制

- 工作时段：由模块负责人一线响应，项目经理二线协调。  
- 非工作时段：值班工程师一线响应，安全负责人/发布负责人二线值守。  
- 值班表更新频率：每周更新一次并归档。  
- 交接要求：值班交接必须包含“未闭环问题清单 + 当前风险等级”。

`<项目>-<模块>-<文档类型>-<版本>-<YYYYMMDD>.md`

### 版本策略

- 主版本（V1.0 -> V2.0）：流程或模板结构性变更。

## 11. 新人上手执行顺序（首月）

### 第 1 周
- 跟做 1 个需求（只做需求与设计环节）。  
- 完成 1 次 AI 输入审批与脱敏演练。  

### 第 2 周
- 主导 1 个小功能全流程（含客户确认）。  
- 输出完整 PRD 与映射表。  

### 第 3 周
- 参与开发测试联动，完成一次缺陷闭环。  
- 独立输出发布说明与复盘草稿。  

### 第 4 周
- 复盘首月执行偏差，更新模板。  
- 通过负责人评估后转入独立负责。
