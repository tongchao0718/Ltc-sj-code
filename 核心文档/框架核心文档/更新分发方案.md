## 1. 方案概述

### 1.1 目标

- **开发者**：在当前仓库执行一条命令，生成可 zip 分发的更新包（版本取自 `package.json`）。
- **用户**：解压后在项目根目录执行 **一条命令**；**仅当更新包版本大于现网版本时才安装**，否则自动跳过。
- **子应用保护**：更新包**不删除**用户自建的 `src/apps/<应用>/` 与 `src/docs/子应用文档/<应用>/`；`subApps.js` 自动合并登记项；自建路由与嵌入前缀**自动合并**（§14.3）。
- **零风险闭环**：全量备份 → 合并 → 注册表 + **子应用入口健康**（§3.7）+ 主应用构建 → **任一关键步骤失败则自动回滚**（§14.4）。

### 1.4 版本判定规则（核心）


| 比较结果                                  | 行为                   |
| ------------------------------------- | -------------------- |
| 更新包版本 **>** 现网 `package.json.version` | 执行合并更新               |
| 更新包版本 **=** 现网版本                      | 提示「已是最新」，**不修改任何文件** |
| 更新包版本 **<** 现网版本                      | 提示「跳过降级」，**不修改任何文件** |


- **现网版本**：项目根 `package.json` → `version`（语义化版本 `major.minor.patch`）。
- **更新包版本**：`update-info.json` → `version`；缺失时回退读 `files/package.json` → `version`。
- **不硬编码**目标版本号；维护人只需 bump `package.json` 后重新 `npm run create-update-package`。
- **最低兼容版本**（可选）：`update-info.json` → `minCompatibleVersion`；仅当现网版本**低于**该值时拒绝升级（防止跨度过大的跳跃），与「大于才更新」独立生效。

### 1.2 两种更新路径（勿混用）


| 路径             | 命令                  | 适用场景               | 说明                                                 |
| -------------- | ------------------- | ------------------ | -------------------------------------------------- |
| **离线更新包（推荐）**  | 包内 `node update.js` | 1.0 无 Git、内网/U 盘分发 | 合并框架，保留自建子应用                                       |
| **Git 拉取（可选）** | `npm run update`    | 同一 Git 仓库协作        | 执行 `git pull` + `npm install` + `build`；**不**替代离线包 |


> `npm run update` 实现为 `AIEP-WEB/src/scripts/update.js`，默认 `git pull origin main`；本仓库主分支为 `master` 时需自行调整或改用离线包。

### 1.3 仓库路径约定

> **文件夹名可任意**：克隆后可叫 `AIEP-DEV`、`Ltc-sj-code`、`my-ltc` 等；更新脚本**只认目录结构**，不认文件夹名或 `package.json.name`。

```
{项目文件夹}/                ← 仓库根（文件夹名随意；须含 package.json + AIEP-WEB/）
├── package.json             ← 版本真值（§1.4）；name 可为 ltc-demo 或团队自定义
├── .cursor/                 ← Agent Skills + Rules（离线包 §2.3 已打入）
│   ├── skills/              ← 总编排 ai-full-process-design、域/移动规范等
│   └── rules/               ← 子应用推进时自动触发规则
├── scripts/                 ← sync-skills（postinstall 自动）、sync-cursor/trae-skills
├── AIEP-WEB/
│   ├── src/config/subApps.js    ← 子应用注册表（单一真值）
│   ├── src/router/index.js      ← 嵌入路由
│   ├── src/apps/<folder>/       ← 子应用源码
│   └── src/docs/子应用文档/     ← 各子应用 SDD/PRD/门禁
├── AIEP-SERVER/             ← 后端（可选）
└── 核心文档/
    ├── 框架核心文档/        ← 框架开发、打包、接入、本文档
    ├── AI+产品落地/         ← 产品流程与填报模板（最新规范）
    └── 执行说明.txt         ← 接手后 5 阶段检查清单
```

#### 1.3.1 旧版 flat-web 自动迁入（2.1.0+）

1.x 工程若 `src/`、`vite.config.js` 在**根目录**（无 `AIEP-WEB/`），`update.js --yes` 在备份后会**自动**：


| 原路径（flat-web）                 | 迁入后（monorepo）            |
| ----------------------------- | ------------------------ |
| `src/`                        | `AIEP-WEB/src/`          |
| `build/`、`dist/`、`public/`    | `AIEP-WEB/` 下同名          |
| `vite.config.js`、`index.html` | `AIEP-WEB/`              |
| `scripts/`（除 sync-skills*）    | `AIEP-WEB/scripts/`      |
| 根 `docs/`                     | `AIEP-WEB/src/docs/`（合并） |
| 根 `install-deps.js` 等         | `AIEP-WEB/src/tools/` 等  |


`scripts/sync-skills.mjs` 等仍留在**仓库根** `scripts/`。`package.json` 脚本恢复 `cd AIEP-WEB &&` / `node AIEP-WEB/scripts/...`。

**子应用零损**：`src/`、`build/` 为**整目录 rename**，不拆分；`src/apps/<folder>/`、`src/docs/子应用文档/`、`subApps.js`、自建 `build/vite.<folder>.config.js` 与 `dev:/build:<folder>` 脚本一并迁入并自动改前缀；合并框架时仍**跳过覆盖**自建子应用目录，并从 flat-web 备份恢复自建路由。

**根目录不再显示迁余**：迁入完成后自动删除根目录残留的 `src/`、`build/`、`dist/`、`vite.config.js`、`docs/` 等（仅当 `AIEP-WEB/` 内已有对应副本时删除，避免误删）。

**回滚与续跑（2.1.0+）**：

- flat-web 项目**自动回滚**时，除恢复备份路径外，会**删除**本次更新产生的 `AIEP-WEB/`（避免与根目录 `src/` 并存导致布局误判）。
- 若备份清单已写入（`backups/v*/update-backup-manifest.json`）但合并未完成，可用 **`--reuse-backup`** 跳过重新备份、从合并阶段续跑。
- 再次更新前若发现「根目录仍有 `src/` 且存在残留 `AIEP-WEB/`」，脚本会先清理残留 `AIEP-WEB/` 再迁入。
- 迁入 monorepo 后，`package.json` 的 `dev` / `build` / `preview` 等会强制同步为 `cd AIEP-WEB && vite ...`（避免回滚后仍为根目录 `vite build` 导致找不到 `index.html`）。

保持扁平结构（不创建 `AIEP-WEB/`）：`node update.js --yes --keep-flat-web`。

## 2. 开发者端：一键打包

### 2.0 Agent 自然语言怎么说（规范）

> 完整命令与自然语言对照见文首 **「分发命令速查」**；本节保留框架更新侧重说明。

在 **Cursor / TRAE Agent 对话**中，**无需**手写完整 npm 命令；Agent 按意图选用下表命令。

#### 2.0.1 维护方：生成更新分发包


| 意图        | 对 Agent 怎么说                | Agent 将执行                                              |
| --------- | -------------------------- | ------------------------------------------------------ |
| 打框架更新包    | 「**生成更新分发包**」「**创建离线更新包**」 | `npm run create-update-package`                        |
| 指定版本后打包   | 「bump 版本后**生成更新包**」        | 先改根 `package.json` `version`，再 `create-update-package` |
| 仅目录不打 zip | 「生成更新包 **不要 zip**」         | `npm run create-update-package -- --no-zip`            |


**产物**（默认目录 + zip）：

- `dist/ltc-demo-update-v{version}/`（`{version}` = 根 `package.json.version`）
- `dist/ltc-demo-update-v{version}.zip`（默认自动压缩）

#### 2.0.2 用户方：安装更新包


| 意图   | 对 Agent 怎么说                     | Agent 将执行                                                                               |
| ---- | ------------------------------- | --------------------------------------------------------------------------------------- |
| 预览升级 | 「**预览框架更新**」「update.js dry-run」 | `node ltc-demo-update-v*/update.js --dry-run`（**可在 Agent 内执行**）                         |
| 确认安装 | 「**更新框架**」「**安装更新包** --yes」     | **禁止** Agent 在 IDE/Trae 沙箱内执行；须提示用户在本机 **PowerShell/CMD** 运行 `node .../update.js --yes` |
| 回滚   | 「**回滚更新**」                      | 同上，须本机终端：`node update.js --rollback backups/v*-<时间戳>`                                   |


> **沙箱拦截**：`update.js` 检测到 Trae `trae-sandbox`、Cursor 沙箱等环境时，会**拒绝** `--yes` / `--rollback`（避免 Windows 退出码 `-1073740791` 等进程崩溃）。`--dry-run` 不受限。CI 等特殊环境可加 `--allow-sandbox` 或 `LTC_ALLOW_SANDBOX=1`。

#### 2.0.3 与 pack / export 的区分（必记）


| 用户意图              | 走哪条                                       | 怎么说                     | 产物                         |
| ----------------- | ----------------------------------------- | ----------------------- | -------------------------- |
| **框架升级**（保留自建子应用） | 本方案 `create-update-package` → `update.js` | 「**更新框架**」「**生成更新分发包**」 | `dist/ltc-demo-update-v*/` |
| **子应用静态演示**       | `pack:sub-app`                            | 「**打包子应用** xxx」         | `AIEP-WEB/dist/{folder}/`  |
| **单个子应用源码交接**     | `export:sub-app`                          | 「**导出子应用** xxx（源码）」     | `dist/sub-app-bundles/`    |


**易混**：

- 「更新 marketing-demo」→ 可能是 **import 子应用 bundle**，不是框架 `update.js`
- 「把整个项目给同事最新框架」→ **更新分发包**，不是 export 单个子应用

**交叉引用**：本文 **§4** ·《子应用打包指南》§9.0.2 · `export-import-sub-app` Skill

### 2.3 打入更新包的范围


| 类别           | 是否打入  | 规则                                                                                            |
| ------------ | ----- | --------------------------------------------------------------------------------------------- |
| 主应用壳         | 是     | `App.vue`、`views/`、`router/`、`config/`、`style.css` 等                                          |
| 框架子应用        | 是     | 仅 `subApps.js` 中登记的 `folder`（如 `sample-app`、`ai-smart-crm-admin`）                             |
| 用户自建子应用      | **否** | `src/apps/` 下不在注册表中的目录不打入、安装时不删除                                                              |
| 子应用文档        | 部分    | 仅框架子应用对应 `docs/子应用文档/<appCode>/`；用户文档保留                                                       |
| 框架核心文档       | 是     | `核心文档/框架核心文档/` 整目录覆盖合并                                                                        |
| AI+产品落地      | 是     | `核心文档/AI+产品落地/` 模板与流程文档                                                                       |
| 执行说明         | 是     | `核心文档/执行说明.txt`                                                                               |
| AIEP-SERVER  | 是     | `app.js`、`src/main/`、框架子应用 `src/apps/<folder>/`；**不删除**用户自建后端子应用                              |
| Agent Skills | 是     | `.cursor/skills/`（总编排 + 域/移动规范）、`AIEP-WEB/src/skills/`（六步阶段 Skill，随 `src/` 打入）                |

## 3. 用户端：一键更新

### 3.1 前置条件

- Node.js **>= 18**
- 目标项目为 **AIEP 仓库根**：含 `AIEP-WEB/` 与根 `package.json`（`name: ltc-demo` 或含 `validate:sub-app-registry` 等脚本）
- 更新包版本须**严格大于**现网 `package.json.version`（见 §1.4）

### 3.2 操作步骤（零风险：先预览再确认）

**最简单（无需记命令）**：将 `ltc-demo-update-v{version}/` 解压到**项目根目录**后：

1. 双击 **`预览更新.bat`** — 只看计划，不改文件（日志：`预览更新-日志.txt`）
2. 双击 **`安装更新.bat`** — 输入 `Y` 确认安装（实时进度 + `安装更新-日志.txt`；详细日志另见 `backups/v*/update.log`）

> 双击后会新开黑色窗口并保持打开；若闪退，请打开同目录 `*-日志.txt` 查看原因。

**命令行**（与 .bat 等价）：

**推荐**：将 `ltc-demo-update-v{version}/` 解压到**项目根目录**（与 `package.json` 同级），**两步执行**：

```bash
# 1. 预览计划（不修改任何文件）
node ltc-demo-update-v{version}/update.js --dry-run

# 2. 确认安装（失败自动回滚）
node ltc-demo-update-v{version}/update.js --yes
```

或进入更新包目录执行（脚本会**向上查找**项目根，并在**同级目录**自动探测相邻项目文件夹）：

```bash
cd ltc-demo-update-v{version}
node update.js --dry-run
node update.js --yes
```


| 参数                     | 作用                                    |
| ---------------------- | ------------------------------------- |
| `--dry-run`            | 仅显示将执行的步骤与自建子应用前缀，**不写盘**             |
| `--yes` / `-y`         | 跳过确认提示，执行安装（**必须**显式添加才会改文件）          |
| `--strict`             | Git 工作区有未提交修改时**拒绝**继续                |
| `--rollback <备份目录>`    | 从 `backups/v{version}-*/` 一键回滚        |
| `--reuse-backup <目录>`  | 复用已有备份目录，**跳过重新备份**，从合并/迁入阶段续跑（须含 `update-backup-manifest.json`） |
| `--project` / `-p`     | 指定项目根相对路径                             |
| `--keep-flat-web`      | 保持 flat-web 扁平结构，不创建 `AIEP-WEB/`        |
| `--verify-sub-apps`    | 更新后直接并行验证子应用（不询问）                     |
| `--no-verify-sub-apps` | 更新后不询问、直接跳过子应用验证                      |
| `--skip-install`       | 跳过 `npm install`（依赖未变时可手动加速）          |
| `--skip-main-app-build` | 跳过主应用 `build` 校验（须已手动 `npm run build` 且 `AIEP-WEB/dist/index.html` 存在） |
| `--delete-package`     | 更新成功后**不询问**，自动删除更新包目录及同目录 `{包名}.zip` |
| `--keep-package`       | 更新成功后**不询问**，保留更新包（跳过删除确认）            |
| `--keep-all-backups`   | 更新成功后**不删除** `backups/` 下历史目录（默认仅保留当次备份） |


**历史备份清理（默认）**：更新**成功**后，`backups/` 下旧的 `v{version}-<时间戳>/` 会自动删除，**仅保留当次备份**供 `--rollback`。需留多份历史时加 `--keep-all-backups`。

**更新包清理（默认交互）**：主应用验证通过并打印 §13 清单后，终端询问「是否删除更新包 `ltc-demo-update-v`* 及同目录 zip？」；确认后于进程退出后自动删除（Windows 无法删除正在执行的 `update.js` 所在目录，故延迟数秒）。非交互终端默认保留，可加 `--delete-package`。

**子应用验证（默认交互）**：主应用通过后，终端列出可验证子应用并询问「是否立即验证？」；输入 `y` 即并行执行全部 `build:<folder>`，**无需手输**各子应用命令。

**耗时说明**（团队反馈「更新要一小时」时对照终端 `⏱` 行）：


| 阶段            | 典型耗时     | 说明                                           |
| ------------- | -------- | -------------------------------------------- |
| 备份 + 合并       | 1～3 分钟   | 复制 `AIEP-WEB/src`、核心文档等                      |
| `npm install` | 5～30+ 分钟 | 内网/首次安装最慢；**dependencies 未变时自动跳过**           |
| 主应用 `build`   | 2～8 分钟   | 必须通过，失败会回滚                                   |
| 子应用并行 `build` | 数～数十分钟   | **更新成功后询问确认**；拒绝则跳过；`--verify-sub-apps` 强制验证 |


推荐团队命令：`node update.js --yes`（约 5～15 分钟；末尾按需 `y` 确认子应用验证）。

更新包不在项目内时：

```bash
node path/to/ltc-demo-update-v{version}/update.js --project .. --yes
```

亦可使用环境变量（相对或绝对均可）：

```bash

set LTC_PROJECT_ROOT=..
node ltc-demo-update-v{version}\update.js --yes

# macOS / Linux
LTC_PROJECT_ROOT=.. node ltc-demo-update-v{version}/update.js --yes
```

### 3.3 安装脚本自动执行项

`update.js`（即 `apply-update-package.mjs`）顺序如下：

1. **预检** — Node ≥ 18；定位项目根；版本比较（§1.4）；检测 Git 脏工作区（`--strict` 时阻断）
2. **确认** — 无 `--yes` 时仅提示计划并退出；`--dry-run` 到此结束
3. **全量备份** → `backups/v{version}-{时间戳}/` + `update-backup-manifest.json`
  - `package.json`、`package-lock.json`
  - `AIEP-WEB/src`、`build`、`scripts`、`vite.config.js`、`index.html`
  - `核心文档/框架核心文档`、`AI+产品落地`、`执行说明.txt`
  - 根目录遗留框架文档（§2.5，存在则备份）
  - `AIEP-SERVER/`（若存在）
4. **合并 `files/`**
  - `package.json`：依赖以包为准；`dev:*` / `build:*` / `validate:*` 脚本**保留用户已有项**
  - `AIEP-WEB/src`：覆盖框架文件；**跳过**用户独有 `apps/`、`docs/子应用文档/` 目录
  - `subApps.js`：**路由稳定后**按 `id` 合并（包内框架条目覆盖 name/desc/to/icon 等 + 保留自建条目）；`pageCount` 与 `router` 可渲染页对齐（应用中心 / 首页指标一致）
  - `build/`：**不覆盖**用户自建 `vite.<folder>.config.js`（§14.3）
  - **深度同步**框架 `src/apps/<folder>/`（整目录 merge，补全缺失 `.vue`）
  - **同步**框架子应用顶层路由块；补全 router 引用但磁盘缺失的页面；裁剪更新包中也不存在的路由
  - **子应用入口健康检查**（`sub-app-route-health.mjs`）：壳组件、入口 `to`、子路由非空、`.vue` 落地 — 失败则阻断更新（**细则见 §3.7**）
  - **自动合并**自建子应用 `router/index.js` 路由块（§14.3）
5. 合并 `核心文档/AI+产品落地`、`执行说明.txt`、`AIEP-SERVER`
6. **根目录文档迁入** — §2.5：同名比对合并至 `核心文档/`，删除根目录副本（保留 `问答场景0527.md`）
7. **清理废弃文档** — 删除 §2.4 所列 1.0 遗留文件（存在则删）
8. `npm install`（根目录 + `AIEP-SERVER/` 若存在）
9. **主应用验证**（§3.6）— 注册表 + `npm run build` + 产物检查；**异常则输出说明并自动回滚**
10. **子应用验证**（§3.6）— 默认**交互确认**后并行 `build:<folder>`；`--verify-sub-apps` / `--no-verify-sub-apps` 可跳过询问；**异常仅提示，不回滚**
11. 终端打印 **§13** 熟悉项目与规范清单；输出 `--rollback` 命令备查

### 3.4 更新后手动抽检（可选）

安装脚本已自动完成主/子应用构建验证。若需人工确认运行时行为，可再执行：

```bash
npm run dev
npm run validate:sub-app-registry
```

子应用构建若在安装时已提示异常，按终端「异常说明」修复后执行 `npm run build:<folder>`。

### 3.5 更新后必读（自动提示）

`update.js` 安装成功后会**在终端打印** §13 的精简清单（文档路径与命令）。完整说明见下文 **§13 更新后熟悉项目与规范执行**。

### 3.6 更新后自动验证（主应用阻断 / 子应用提示）


| 对象      | 验证项                                                    | 失败行为                                    |
| ------- | ------------------------------------------------------ | --------------------------------------- |
| **主应用** | `validate:sub-app-registry`（含子应用入口健康：防空白页）             | 输出 JSON `failed_items` 等异常说明 → **自动回滚** |
| **主应用** | `npm run build` + `AIEP-WEB/dist/index.html` 存在        | 输出构建日志尾部 → **自动回滚**                     |
| **子应用** | 交互确认或 `--verify-sub-apps` 时并行 `npm run build:<folder>` | 输出该子应用构建异常说明 → **仅提示，更新仍算成功**           |
| **子应用** | 无 `build:<folder>` 脚本或源码目录缺失                           | 标记「跳过」并说明原因                             |


**原则**：只有主应用异常会阻断更新；子应用问题由用户按终端说明自行修复，不影响框架升级结果。

### 3.7 子应用合并、注册表同步与入口健康（v2.1.0+）

> **背景**：更新后若仅合并 `router/index.js` 而未同步 `src/apps/` 下页面文件，或路由裁剪后子路由为空，会出现「能进子应用但页面空白」。应用中心、首页指标若与 `subApps.js` 不一致，则入口 `to` 与统计数字会误导用户。本节为 **§3.3 第 4 步** 的展开说明。

#### 3.7.1 合并顺序（`update.js` 内固定）


| 顺序  | 动作                              | 说明                                                 |
| --- | ------------------------------- | -------------------------------------------------- |
| 1   | `mergeSrcTree`                  | 覆盖框架 `src/`（**跳过**用户独有 `apps/`、`docs/子应用文档/`）      |
| 2   | **深度同步**框架 `src/apps/<folder>/` | 从 `files/AIEP-WEB/src/apps/` 整目录 merge，补全缺失 `.vue` |
| 3   | 合并自建路由                          | 从备份提取非框架前缀的顶层路由块追加到 `router/index.js`（§14.3）       |
| 4   | **同步**框架顶层路由块                   | 用更新包 `router/index.js` 中对应 `/前缀` 块覆盖现网（保留自建前缀）     |
| 5   | 补全缺失页面                          | 扫描 router 中 `import('…vue')`，从 `files/` 复制磁盘缺失文件   |
| 6   | 裁剪无效路由                          | 移除更新包中也不存在的页面路由（避免 build 失败）                       |
| 7   | **同步 `subApps.js`**             | 路由稳定后：框架条目以包为准 + 保留自建条目 + `pageCount` 与 router 对齐  |
| 8   | **入口健康检查**                      | 合并末尾 + `validate:sub-app-registry` 再次校验            |


#### 3.7.2 `subApps.js` 与应用中心 / 首页的关系（单一真值）


| 消费方   | 路径                                 | 读取内容                                                                                 |
| ----- | ---------------------------------- | ------------------------------------------------------------------------------------ |
| 应用中心  | `AIEP-WEB/src/views/AppCenter.vue` | `subApps` 全量列表（名称、图标、`to`、描述、tag）                                                    |
| 首页工作台 | `AIEP-WEB/src/views/Dashboard.vue` | `subApps` + `getSubAppMetrics()`（应用数、页面数、快速入口）                                       |
| 主应用嵌入 | `AIEP-WEB/src/App.vue`             | `isSubAppEmbedPath` / `getSubAppEmbedPrefixes()`（**禁止**硬编码 `SUB_APP_EMBED_PREFIXES`） |


**更新后自动保障**：

- 框架子应用：`name`、`desc`、`icon`、`to`、`tag`、`pageCount` 等以更新包 `files/AIEP-WEB/src/config/subApps.js` 为基准，按 `id` 覆盖现网框架条目。
- 自建子应用：按 `id` **追加保留**，不被包内条目删除。
- `pageCount`：对框架子应用，按 `router/index.js` 中该前缀下**可 lazy-load 的 `.vue` 页数**（不含顶层壳组件）自动回写，使首页「页面数量」与路由一致。

#### 3.7.3 入口健康检查项（`sub-app-route-health.mjs`）

对每个 `subApps.js` 登记项执行：


| 检查项                                   | 失败后果                 |
| ------------------------------------- | -------------------- |
| `router` 存在对应顶层 `path: '/<folder>'`   | 应用中心点击无法进入           |
| 顶层**壳组件** `.vue` 存在                   | 子应用区域空白              |
| 子路由中至少 1 个可渲染页（非仅 redirect）           | 进入后 `router-view` 空白 |
| 登记项 `to` 在路由块中有匹配 `path` 或 `redirect` | 应用中心默认入口失效           |
| 入口目标 `.vue` 文件存在                      | 首屏空白                 |
| `src/apps/<folder>/` 目录存在             | 嵌入/构建异常              |


**门禁**：合并阶段 `assertSubAppsRegistryHealthy` 失败 → 错误码 `MAIN_APP_SUBAPP_BLANK_RISK` → **自动回滚**；`npm run validate:sub-app-registry` 含同一套检查。

#### 3.7.4 已更新后出现空白时的修复

不重新跑完整 `--yes`（版本相等会跳过）时，使用 `**--repair-routes`**：

```bash
node ltc-demo-update-v{version}/update.js --repair-routes --dry-run
node ltc-demo-update-v{version}/update.js --repair-routes --yes
npm run build
npm run validate:sub-app-registry
```

该模式执行 §3.7.1 中第 2～8 步（不修改 `package.json` 版本），适用于 router 已变但 `apps/` 未落地或 `subApps.js` 未同步的历史环境。

## 4. 子应用源码导出/导入（与框架更新区分）

> **定位**：交接**单个子应用源码**（对方在另一份 ltc-demo 仓库继续开发），**不是**框架 `update.js`，也**不是** `pack:sub-app` 静态 dist。  
> **细则真值**：《子应用打包指南》§9 · `AIEP-WEB/src/skills/export-import-sub-app/SKILL.md`

### 4.1 三种交接路径（勿混用）

| 用户意图 | 走哪条 | 怎么说（Agent） | 产物 |
| --- | --- | --- | --- |
| **框架升级**（保留自建子应用） | `create-update-package` → `update.js` | 「更新框架」「生成更新分发包」 | `dist/ltc-demo-update-v{version}/` |
| **子应用静态演示 / 部署** | `pack:sub-app` | 「打包子应用 xxx」「导出静态页」 | `AIEP-WEB/dist/{folder}/` |
| **单个子应用源码交接** | `export:sub-app` → `import:sub-app` | 「**导出子应用** xxx（源码）」 | `dist/sub-app-bundles/{app-code}-bundle-{日期}/` |

**易混反例**：

| 错误理解 | 正确做法 |
| --- | --- |
| 「导出子应用给客户看演示」 | 走 **pack:sub-app**（静态 dist），不是 export |
| 「打包源码交给同事改代码」 | 走 **export:sub-app**（源码 bundle），不是 pack |
| 「更新 marketing-demo」 | 可能是 **import 子应用 bundle**，不是框架 `update.js` |
| 「把整个项目给同事最新框架」 | 走 **更新分发包**，不是 export 单个子应用 |

### 4.2 导出规则（A 方）

**执行位置**：仓库根目录（含根 `package.json`）。

**app-code 与 folder**：`--app {app-code}` 从 `subApps.js` 解析 `folder`；二者不一致时以注册表为准（如 `ai-smart-crm` → `ai-smart-crm-admin`），**说 app-code**。

**导出前须满足**（否则脚本失败）：

| 条件 | 说明 |
| --- | --- |
| `subApps.js` 已登记 | 含该 `id` / `appCode` |
| `router/index.js` 已接入 | 存在与 `to` 前缀对应的顶层路由块 |
| `src/apps/{folder}/` 存在 | 子应用源码目录在盘 |

**命令**：

```bash
npm run export:sub-app -- --app {app-code}
npm run export:sub-app -- --app marketing-demo --zip    # 默认打 zip，便于 U 盘
npm run export:sub-app -- --app {app-code} --no-zip     # 仅目录
npm run export:sub-app -- --app {app-code} --out dist/my-handoff   # 自定义输出目录
```

**产物根路径**：`dist/sub-app-bundles/{app-code}-bundle-{YYYY-MM-DD}/`（或 `--out` 指定路径）。

### 4.3 bundle 目录结构（目录规则）

将 **整个 bundle 文件夹**（或 zip 解压后）交给接收方；勿只拷贝 `files/` 子目录。

```text
dist/sub-app-bundles/{app-code}-bundle-{日期}/
├── files/                              # 待写入目标仓库的文件树
│   ├── AIEP-WEB/src/apps/{folder}/     # 子应用源码（必选）
│   ├── AIEP-WEB/src/docs/子应用文档/{app-code}/   # 子应用文档（若有）
│   ├── AIEP-WEB/build/vite.{folder}.config.js     # 独立构建配置（若有）
│   ├── AIEP-WEB/build/{folder}.html               # 独立入口 HTML（若有）
│   └── AIEP-SERVER/src/apps/{folder}/             # 后端子应用（若有）
├── manifest.json                       # subApps 条目、router 前缀、package scripts、files 清单
├── integration/
│   └── router-block.js                 # 从 router/index.js 提取的顶层路由块
├── integrate.mjs                       # 接收方可独立运行的一键集成（不依赖框架 import 脚本）
└── README.md                           # 交接说明与命令
```

| 路径 | 规则 |
| --- | --- |
| `files/` | 导入时按相对路径合并到目标仓库；**不得**手工改 manifest 内路径约定 |
| `manifest.json` | 导出脚本生成；含 `subAppEntry`、`routerPrefix`、`packageScripts` |
| `integration/router-block.js` | 与主应用 `router/index.js` 中该前缀块一致；导入时追加或 `--force` 覆盖 |
| `integrate.mjs` | U 盘 / 未含 `import:sub-app` 的旧框架时，在 bundle 目录执行即可 |

### 4.4 导入规则（B 方）

**执行位置**：目标 ltc-demo **仓库根目录**（bundle 本身可放在任意路径）。

```bash
# 预览（不写盘）
npm run import:sub-app -- --bundle path/to/{app-code}-bundle-{日期} --dry-run

# 确认写入
npm run import:sub-app -- --bundle path/to/bundle --yes

# 接收方框架较旧、无 import 脚本时
node path/to/bundle/integrate.mjs --yes
```

**导入冲突**（仓库已有同名子应用时）：

| 参数 / 选择 | 行为 |
| --- | --- |
| 默认 / 取消 | **不写入**；exit 2 并提示，避免 silent partial 导入 |
| `--force` | 覆盖源码 + `subApps.js` + `router/index.js` + `package.json` 中 `dev:/build:` 脚本 |
| `--files-only` | **仅**覆盖 `files/` 内源码，不改注册表与路由 |
| `--dry-run` | 仅报告冲突与计划步骤 |

**导入后必做**：

```bash
npm run validate:sub-app-registry
npm run dev:{folder}
npm run dev    # 可选：主应用嵌入区冒烟
```

### 4.5 与框架更新（§3、§14）的关系

| 场景 | 用哪条 |
| --- | --- |
| 只交一个子应用给同事继续开发 | **§4** export → import |
| 同事已有 ltc-demo，你要给他**最新框架**且保留他自建子应用 | **§3** `update.js` |
| 只交可部署静态页、不看源码 | `pack:sub-app`（《子应用打包指南》§4） |

框架 `update.js` **不会删除**通过 import 接入的自建子应用（`folder` 不在 `framework-meta.json` 的 `frameworkAppFolders` 内）；详见 **§14.3**。  
若子应用已在 A 仓开发完毕，B 仓仅需接入：优先 **§4 import**，不必等 B 仓先跑框架更新。

## 5. 更新包元数据

### 5.1 update-info.json（字段说明）


| 字段                     | 必填  | 说明                                         |
| ---------------------- | --- | ------------------------------------------ |
| `version`              | 是   | 与打包时 `package.json.version` 一致；**用于与现网比较** |
| `releaseDate`          | 是   | 发布日期 ISO 日期                                |
| `updateType`           | 是   | 固定 `framework`                             |
| `updateMode`           | 是   | 固定 `merge`                                 |
| `preserveUserSubApps`  | 是   | 固定 `true`                                  |
| `minCompatibleVersion` | 否   | 现网低于此版本时拒绝升级（可选）                           |
| `frameworkAppFolders`  | 是   | 框架子应用目录列表                                  |
| `obsoletePaths`        | 是   | 更新后必须删除的废弃文档相对路径列表（§2.4）                   |
| `requirements`         | 否   | Node/npm 要求                                |


打包脚本自动生成，无需手写目标版本。

### 5.2 framework-meta.json（位于 files/）

供 `update.js` 判断哪些 `apps/`、`docs/子应用文档/` 子目录属于框架、可覆盖更新。

## 6. 自建子应用与合并边界

### 6.1 更新后通常无需改动（脚本自动保障）

- `src/apps/<你的应用>/` 源码 — **不删除、不覆盖**
- `src/docs/子应用文档/<你的应用>/` 文档 — **不删除、不覆盖**
- `package.json` 中已有的 `dev:<folder>`、`build:<folder>` 脚本 — **保留**
- `router/index.js` 中自建子应用路由块 — **从备份自动合并**（§14.3）
- `App.vue` 嵌入模式 — **由 `subApps.js` 的 `isSubAppEmbedPath` 驱动**（v2.1.0+ 不再合并 `SUB_APP_EMBED_PREFIXES`）
- `build/vite.<folder>.config.js`（自建）— **同名不覆盖**（§14.3）

### 6.2 极少数需人工介入的情况


| 情况                      | 处理                                  |
| ----------------------- | ----------------------------------- |
| 自建子应用路由结构非标准（脚本无法解析路由块） | 从 `backups/` 手工合并 `router/index.js` |
| 更新前注册表校验已失败（项目本身未接好）    | 先按接入规范修好，再执行更新                      |
| 更新后需验证自建子应用构建           | 手动执行 `npm run build:<folder>`       |


> 注册表校验失败时，安装脚本会**自动回滚**，不会留下半成品。

### 6.3 禁止依赖的行为

- **不要**对 `src/` 做整目录替换（旧方案会删除 `apps/`）。
- **不要**在未备份时手动覆盖 `src/apps/`。
- **不要**省略 `--yes` 后误以为已更新（无 `--yes` 默认不写盘）。

## 7. 回滚

### 7.1 自动回滚（默认）

`validate:sub-app-registry` 或 `npm run build` 失败时，`update.js` 会**立即**从当次备份目录恢复，项目回到更新前状态。

### 7.2 手动回滚

```bash
node update.js --rollback backups/v{version}-{时间戳}
npm install
npm run build
```

备份目录含 `update-backup-manifest.json`，记录当次覆盖的路径列表与自建子应用前缀。

### 7.3 手工文件恢复（兜底）

若 `--rollback` 不可用，将 `backups/v{version}-*/` 内文件按目录结构拷回项目根，再执行 `npm install && npm run build`。

## 8. 故障处理

### 8.1 找不到项目根（最常见）

**现象**：`❌ 未找到 AIEP 项目根目录`

**原因**：


| 原因                             | 说明                                                  |
| ------------------------------ | --------------------------------------------------- |
| 误以为文件夹须叫 `AIEP-DEV`            | **不要求**；`Ltc-sj-code`、`project-code-package_gx` 等均可 |
| 旧版 **flat-web**（无 `AIEP-WEB/`） | v2.1.0+ 支持；**可无 subApps.js**（1.x 旧工程），更新时自动创建       |
| 只在 `AIEP-WEB/` 或 `src/` 子目录执行  | 须在**项目根**（含根 `package.json`）执行                      |
| 更新包与项目**同级**解压                 | v2.1.0+ 会自动探测同级项目；或 `--project ../你的项目文件夹名`         |
| `package.json` 的 `name` 被改过    | v2.1.0+ 已放宽：认 `AIEP-WEB` 结构指纹，不限 `ltc-demo`         |
| Cursor 只打开了子目录                 | 应打开含 `AIEP-WEB/`、`核心文档/` 的完整仓库根                     |


**正确目录结构示例**：

```text
方式 A（推荐）：更新包在项目根内
Ltc-sj-code/                 ← 文件夹名示例，可任意
├── package.json
├── AIEP-WEB/
└── ltc-demo-update-v2.1.0/
    └── update.js

方式 B：更新包与项目同级（Desktop 常见）
Desktop/
├── Ltc-sj-code/             ← 项目（名字随意）
└── ltc-demo-update-v2.1.0/  ← 更新包
```

**处理**：

```bash
# 标准 monorepo（含 AIEP-WEB/）
node ltc-demo-update-v2.1.0/update.js --dry-run

# 旧版 flat-web（src/ 在根目录，如 project-code-package_gx/）
cd project-code-package_gx   # 或 --project 指向该目录
node ../ltc-demo-update-v2.1.0/update.js --dry-run
node ../ltc-demo-update-v2.1.0/update.js --yes
# 脚本会提示：检测到旧版扁平结构（flat-web 模式）
```

**方式 B** — 更新包与项目同级：

```bash
cd ltc-demo-update-v2.1.0
node update.js --project ../Ltc-sj-code --dry-run
node update.js --project ../Ltc-sj-code --yes
# 若同级只有一个项目，v2.1.0+ 也可能自动识别
```


| 现象                                           | 处理                                                                                |
| -------------------------------------------- | --------------------------------------------------------------------------------- |
| 提示「确认继续？添加 --yes」                            | 正常：先 `--dry-run` 预览，再 `--yes` 执行                                                  |
| 提示「已是最新」/「跳过降级」                              | 正常：更新包版本未高于现网，无需操作                                                                |
| 找不到项目根                                       | 见 §8.1：须打开**仓库根**（含 `package.json` + `AIEP-WEB/`）；更新包与项目同级时用 `--project ../项目文件夹` |
| 更新中途失败并提示「已自动回滚」                             | 项目已恢复；修复根因后重新 `node update.js --yes`，或 `--reuse-backup backups/v*-<时间戳>` 续跑 |
| 主应用 build：`Could not resolve entry module "index.html"` | 多为 flat-web 回滚后残留 `AIEP-WEB/` 且 `build` 仍为根目录 `vite build`；用新版包重跑或手动删残留 `AIEP-WEB/` 后再更新 |
| `validate:sub-app-registry` 失败（更新前）          | 先按接入规范修好现网，再更新                                                                    |
| 应用中心看不到自建子应用                                 | 检查 `subApps.js` 是否有你的 `id`；确认更新时终端是否打印了自建前缀                                       |
| `npm run build` 失败                           | 已自动回滚；`npm cache clean --force` 后重试；检查 Node >= 18                                 |
| 更新后子应用页面空白                                   | 见 §3.7.4：`--repair-routes --yes`；日常预防见 §3.7.1～§3.7.3                              |
| Agent 代跑 `--yes` 被拦截或退出码 `3221226505` / `-1073740791` | Agent 仅可代跑 `--dry-run`；正式安装须本人双击 `安装更新.bat` 或在 IDE **终端→新建终端**执行 `--yes`（见 §8.2） |


### 8.2 Agent / IDE 沙箱（Trae、Cursor）

**现象**：

- 备份阶段打印若干 `✓` 后进程异常退出（Windows 退出码 `**-1073740791`** / `0xC0000409`）
- 或启动即提示「检测到 Agent/IDE 沙箱环境，已阻止会修改项目的更新操作」

**原因**：

- Trae `trae-sandbox` 限制 `fs.renameSync` / `fs.rmSync`、大量文件 I/O、`npm`/`PowerShell` 子进程
- 备份/合并/flat-web 迁移文件量大，沙箱内易触发资源或权限上限
- **旧版**沙箱检测曾调用 PowerShell 查父进程，可能在检测阶段即崩溃；**现版已移除**，仅读环境变量与非交互终端

**退出码对照**（均为 Node 整进程被强杀，不是 `catch` 到的 npm 失败）：


| 退出码           | NTSTATUS     | 常见含义                                 |
| ------------- | ------------ | ------------------------------------ |
| `3221226505` / `-1073740791` | `0xC0000409` | Fast Fail / 沙箱或杀毒强杀（**最常见**） |
| `-1073741819` | `0xC0000005` | ACCESS_VIOLATION（非法内存访问）             |


**处理**：


| 操作                         | 在哪里执行                                       |
| -------------------------- | ------------------------------------------- |
| `--dry-run` 预览             | Agent 对话内可代跑                            |
| `--yes` 安装、`--rollback` 回滚 | **本人确认**：双击 `安装更新.bat`（窗口保持打开 + `安装更新-日志.txt`），或 IDE「终端→新建终端」粘贴命令 |
| CI 流水线                     | 加 `--allow-sandbox` 或 `LTC_ALLOW_SANDBOX=1` |


若备份不完整（`backups/` 无 `update-backup-manifest.json`），可删除该目录后重试 `--yes`；**若 manifest 已存在**，优先 `--reuse-backup` 续跑。现网在 manifest 写入后、合并完成前通常未被修改。

**如何判断停在哪一步**（v2.1.0+）：


| 日志 / 文件                                       | 含义                                                               |
| --------------------------------------------- | ---------------------------------------------------------------- |
| `**backups/v*/update.log`**                   | **完整运行日志**（`--yes` 起自动写入；Agent 终端关闭后仍可查）                         |
| 仅有 `✓` 备份行，无 `✓ 备份清单已写入`                      | 备份复制未完成；现网通常未改                                                   |
| 有 `✓ 备份清单已写入` / `update-backup-manifest.json` | 备份关键点已完成；若未看到 `✅ 更新完成`，崩溃在合并/构建阶段                                |
| 停在 `整理备份清单` 且无后续输出                       | **旧版**可能在备份整理处崩溃；**现版**先写 manifest、整理步骤可跳过；manifest 在则可 `--reuse-backup` |
| `backups/.../.update-progress.json`           | 打开 `phase` 字段可见最后阶段（如 `merge` / `npm-install`）                   |
| 退出码 `3221226505` / `-1073740791`             | 多为沙箱/杀毒强杀，**不是**校验失败；改用 `安装更新.bat` 或 IDE 终端重跑 `--yes`                  |


> **说明**：`update.js` 默认不写项目根日志；所有输出镜像到 `backups/v{版本}-{时间戳}/update.log`。Trae/Cursor Agent 面板里的输出**不会**自动保存。

## 9. 安全与分发

- 更新前**全量备份**（§3.3）；关键步骤失败**自动回滚**（§7.1）。
- 建议用户额外复制整个项目目录；有未提交 Git 修改时使用 `--strict` 或先 commit。
- 内网/U 盘分发时，可对 zip 附 `SHA256` 校验（规则见 `创建指南.md` §5.4）。
- 先在测试副本上跑通 `--dry-run` 与 `--yes`，再对主开发目录执行。

## 12. 总结

本方案实现 **「框架可分发、子应用零损、失败可回滚、更新后可按最新规范接手」** 的一键更新，并覆盖 **子应用源码交接**（§4）：

1. 开发者：bump `package.json.version` → `npm run create-update-package` → zip 分发
2. 用户：`--dry-run` 预览 → `node <更新包>/update.js --yes`（**仅当包版本 > 现网版本**）
3. 全量备份 → 合并框架 → **深度同步子应用 + 同步 subApps.js** → 入口健康检查 → 校验门禁 → 构建主应用
4. 校验或构建失败 → **自动回滚**；成功则保留**当次**备份供 `--rollback`（历史备份默认自动清理）
5. 单个子应用交接 → **§4** `export:sub-app` / `import:sub-app`（与 §3 框架更新勿混用）

## 14. 零风险保障机制

> **说明**：软件更新无法做到数学意义上的「绝对零风险」；本方案通过 **预防 → 保护 → 验证 → 回滚** 四层闭环，将可预见风险压到最低，使用户侧感知为「要么成功可用，要么原样退回」。

### 14.1 四层闭环


| 层级     | 机制                                                                     | 效果                      |
| ------ | ---------------------------------------------------------------------- | ----------------------- |
| **预防** | `--dry-run` 预览；无 `--yes` 不写盘；`--strict` 拒绝 Git 脏工作区；版本不大于现网则跳过         | 避免误操作、误降级、误目录           |
| **保护** | 自建 `apps/`、文档目录不删；`subApps.js` 按 id 合并；`build/vite.<自建>.config.js` 不覆盖 | 子应用资产零损                 |
| **验证** | 注册表 + **子应用入口健康**（§3.7）+ 主应用 `build` 作为**完成门禁**                        | 未通过不算更新成功；空白页风险在合并阶段即拦截 |
| **回滚** | 门禁失败自动从当次 `backups/` 恢复；支持 `--rollback`                                | 不留半成品                   |


### 14.2 全量备份范围

每次 `--yes` 安装前写入 `backups/v{version}-{时间戳}/`，并生成 `update-backup-manifest.json`：

- 根 `package.json`、`package-lock.json`
- `AIEP-WEB/` 下 `src`、`build`、`scripts`、入口 HTML/Vite 配置
- `核心文档/框架核心文档`、`AI+产品落地`、`执行说明.txt`
- `.cursor/`（`skills/`、`rules/`）
- `scripts/`（Skill 同步脚本）
- `AIEP-SERVER/`（若存在）

### 14.3 子应用合并与自建保护（核心）

#### 框架子应用（见 §3.7）

- 深度同步 `src/apps/<framework-folder>/`、框架路由块、缺失 `.vue` 补全
- 路由稳定后同步 `subApps.js`（应用中心 / 首页指标 / 嵌入前缀单一真值）
- 合并末尾与子应用注册表校验均执行入口健康检查

#### 自建子应用（零损 + 自动合并）

安装脚本在合并框架版 `router/index.js` 之后，根据备份中 `subApps.js` 识别**非框架**子应用（`folder` 不在 `files/framework-meta.json` 的 `frameworkAppFolders` 内），并：

1. 从备份 `router/index.js` **提取**对应 `/前缀` 的顶层路由对象，追加到现网路由表（若缺失）
2. `**App.vue` 嵌入前缀**：v2.1.0+ 由 `subApps.js` 导出函数驱动，**无需**再手工维护 `SUB_APP_EMBED_PREFIXES`；更新时跳过 App.vue 前缀合并
3. `build/` 合并时，跳过覆盖用户独有的 `vite.<folder>.config.js`
4. `subApps.js` 中自建条目按 `id` **保留**；框架条目由更新包覆盖

因此 §6.2 所列「手工合并 router/App.vue」在标准接入结构下**不再需要**。

### 14.4 失败自动回滚

以下任一步骤抛错即触发回滚（恢复 §14.2 清单内全部路径）：

- 文件合并异常
- `npm install` 失败
- **主应用验证**失败（注册表 / **入口健康** / build / 产物缺失，见 §3.6、§3.7）

**不触发回滚**：子应用 `npm run build:<folder>` 失败（仅终端提示异常说明）。

终端输出：

```text
❌ 更新在「合并/安装/校验/构建」失败: ...
⏪ 正在从备份自动回滚...
✅ 已回滚至更新前状态
```

### 14.5 残余风险与建议


| 残余风险                | 建议                                           |
| ------------------- | -------------------------------------------- |
| 更新包被篡改              | 分发 zip 附 SHA256；内网校验                         |
| 跨大版本 API 不兼容        | 配置 `minCompatibleVersion`；逐级升级               |
| 自建子应用构建未自动执行        | 更新成功后手动 `npm run build:<folder>`             |
| 对 `AI+产品落地` 模板的本地定制 | 会被框架版 merge 覆盖；重要定制应放项目实例文档目录                |
| 更新前项目注册表已损坏         | 先修现网，再更新；否则门禁失败并回滚                           |
| 1.0 废弃文档仍被引用        | 更新后 §2.4 四类文件会被删除；请改读 `安装指南.md` 与 `执行说明.txt` |
