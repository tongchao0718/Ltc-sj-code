# 系统更新分发方案

> **文档定位**：本文档描述**离线更新包**的打包、分发与一键安装机制（以仓库代码为唯一事实来源）。  
> **用户操作步骤**见 `一键更新指南.md`；离线/U 盘交付见 `创建指南.md` §5.4。  
> **子应用接入**见 `主应用子应用接入规范.md`。

**最后对齐**：2026-06-05 · 版本号以根目录 `package.json` 的 `version` 为准；实现见 `create-update-package.js`、`apply-update-package.mjs`。  
**零风险保障**见 **§13**（备份、自动合并自建路由、失败自动回滚、校验门禁）。

---

## 1. 方案概述

### 1.1 目标

- **开发者**：在当前仓库执行一条命令，生成可 zip 分发的更新包（版本取自 `package.json`）。
- **用户**：解压后在项目根目录执行 **一条命令**；**仅当更新包版本大于现网版本时才安装**，否则自动跳过。
- **子应用保护**：更新包**不删除**用户自建的 `src/apps/<应用>/` 与 `src/docs/子应用文档/<应用>/`；`subApps.js` 自动合并登记项；自建路由与嵌入前缀**自动合并**（§13.3）。
- **零风险闭环**：全量备份 → 合并 → 注册表校验 + 主应用构建 → **任一关键步骤失败则自动回滚**（§13.4）。

### 1.4 版本判定规则（核心）

| 比较结果 | 行为 |
|----------|------|
| 更新包版本 **>** 现网 `package.json.version` | 执行合并更新 |
| 更新包版本 **=** 现网版本 | 提示「已是最新」，**不修改任何文件** |
| 更新包版本 **<** 现网版本 | 提示「跳过降级」，**不修改任何文件** |

- **现网版本**：项目根 `package.json` → `version`（语义化版本 `major.minor.patch`）。
- **更新包版本**：`update-info.json` → `version`；缺失时回退读 `files/package.json` → `version`。
- **不硬编码**目标版本号；维护人只需 bump `package.json` 后重新 `npm run create-update-package`。
- **最低兼容版本**（可选）：`update-info.json` → `minCompatibleVersion`；仅当现网版本**低于**该值时拒绝升级（防止跨度过大的跳跃），与「大于才更新」独立生效。

### 1.2 两种更新路径（勿混用）

| 路径 | 命令 | 适用场景 | 说明 |
|------|------|----------|------|
| **离线更新包（推荐）** | 包内 `node update.js` | 1.0 无 Git、内网/U 盘分发 | 合并框架，保留自建子应用 |
| **Git 拉取（可选）** | `npm run update` | 同一 Git 仓库协作 | 执行 `git pull` + `npm install` + `build`；**不**替代离线包 |

> `npm run update` 实现为 `AIEP-WEB/src/scripts/update.js`，默认 `git pull origin main`；本仓库主分支为 `master` 时需自行调整或改用离线包。

### 1.3 仓库路径约定

```
AIEP-DEV/                    ← 项目根（package.json name: ltc-demo）
├── package.json             ← 版本真值（§1.4）
├── .cursor/                 ← Agent Skills + Rules（离线包 §2.3 已打入）
│   ├── skills/              ← 总编排 ai-full-process-design、域/移动规范等
│   └── rules/               ← 子应用推进时自动触发规则
├── scripts/                 ← sync-skills（postinstall 自动）、sync-cursor/trae-skills
├── AIEP-WEB/
│   ├── src/config/subApps.js    ← 子应用注册表（单一真值）
│   ├── src/router/index.js      ← 嵌入路由
│   ├── src/apps/<folder>/       ← 子应用源码
│   └── src/docs/子应用文档/     ← 各子应用 SDD/PRD/门禁
├── AIEP-SERVER/             ← 后端（可选）
└── 核心文档/
    ├── 框架核心文档/        ← 框架开发、打包、接入、本文档
    ├── AI+产品落地/         ← 产品流程与填报模板（最新规范）
    └── 执行说明.txt         ← 接手后 5 阶段检查清单
```

## 2. 开发者端：一键打包

### 2.1 命令

在 **AIEP-DEV 根目录**执行：

```bash
npm run create-update-package
```

等价于：

```bash
node AIEP-WEB/src/scripts/create-update-package.js
```

### 2.2 输出位置

```
dist/ltc-demo-update-v{version}/
├── update.js              # 由 apply-update-package.mjs 复制
├── update-info.json
├── README.md
└── files/                 # 待合并的框架文件
    ├── package.json
    ├── package-lock.json
    ├── framework-meta.json
    ├── AIEP-WEB/
    │   ├── vite.config.js
    │   ├── index.html
    │   ├── build/
    │   ├── scripts/
    │   └── src/           # 不含用户自建子应用
    ├── 核心文档/
    │   ├── 框架核心文档/
    │   ├── AI+产品落地/
    │   └── 执行说明.txt
    ├── .cursor/
    │   ├── skills/          # Agent 总编排与域规范 Skill
    │   └── rules/           # Cursor 项目规则（.mdc）
    ├── scripts/             # sync-cursor-skills.mjs、sync-trae-skills.mjs
    └── AIEP-SERVER/         # 后端（仅框架子应用 apps/）
```

将 `dist/ltc-demo-update-v{version}/` 打成 zip 即可分发（`{version}` 与 `package.json` 一致）。

### 2.3 打入更新包的范围

| 类别 | 是否打入 | 规则 |
|------|----------|------|
| 主应用壳 | 是 | `App.vue`、`views/`、`router/`、`config/`、`style.css` 等 |
| 框架子应用 | 是 | 仅 `subApps.js` 中登记的 `folder`（如 `sample-app`、`ai-smart-crm-admin`） |
| 用户自建子应用 | **否** | `src/apps/` 下不在注册表中的目录不打入、安装时不删除 |
| 子应用文档 | 部分 | 仅框架子应用对应 `docs/子应用文档/<appCode>/`；用户文档保留 |
| 框架核心文档 | 是 | `核心文档/框架核心文档/` 整目录覆盖合并 |
| AI+产品落地 | 是 | `核心文档/AI+产品落地/` 模板与流程文档 |
| 执行说明 | 是 | `核心文档/执行说明.txt` |
| AIEP-SERVER | 是 | `app.js`、`src/main/`、框架子应用 `src/apps/<folder>/`；**不删除**用户自建后端子应用 |
| Agent Skills | 是 | `.cursor/skills/`（总编排 + 域/移动规范）、`AIEP-WEB/src/skills/`（六步阶段 Skill，随 `src/` 打入） |
| Cursor Rules | 是 | `.cursor/rules/`（如 `ai-full-process-design.mdc`） |
| Skill 同步脚本 | 是 | `scripts/sync-skills.mjs`（**postinstall 自动**）、`sync-cursor-skills.mjs`、`sync-trae-skills.mjs` |
| `dist/` 构建产物 | 否 | 用户在本地 `npm run build` 生成 |

框架子应用列表在打包时从 `AIEP-WEB/src/config/subApps.js` 动态读取，写入 `files/framework-meta.json`。

### 2.4 更新后必须删除的废弃文档（强制约束）

自 2.0 起，以下 **1.0 / 过渡版** 文档已废止；`update.js` 在合并完成后**强制删除**（路径存在则删，不存在则跳过）：

| 废弃文件 | 权威替代 |
|----------|----------|
| `环境安装指南.md` | `核心文档/框架核心文档/安装指南.md` |
| `依赖安装指南.md` | `核心文档/框架核心文档/安装指南.md` |
| `A-安装说明.txt` | `核心文档/执行说明.txt` |
| `系统设计规范.md` | `核心文档/框架核心文档/创建指南.md`（§3.3.4）+ `安装指南.md` |

**扫描路径**（相对项目根，清单见 `update-obsolete-paths.mjs` / `update-info.json` → `obsoletePaths`）：

- `核心文档/框架核心文档/<上表文件名>`
- `核心文档/<上表文件名>`（1.0 扁平遗留）
- 项目根下的同名散落文件

**打包侧约束**：上述文件**不会打入**更新包 `files/核心文档/框架核心文档/`（避免合并后再次落盘）。

## 3. 用户端：一键更新

### 3.1 前置条件

- Node.js **>= 18**
- 目标项目为 **ltc-demo** 结构：根目录含 `package.json`（`name: ltc-demo`）与 `AIEP-WEB/`
- 更新包版本须**严格大于**现网 `package.json.version`（见 §1.4）

### 3.2 操作步骤（零风险：先预览再确认）

**推荐**：将 `ltc-demo-update-v{version}/` 解压到**项目根目录**（与 `package.json` 同级），**两步执行**：

```bash
# 1. 预览计划（不修改任何文件）
node ltc-demo-update-v{version}/update.js --dry-run

# 2. 确认安装（失败自动回滚）
node ltc-demo-update-v{version}/update.js --yes
```

或进入更新包目录执行（脚本会**向上自动查找**含 `AIEP-WEB/` 的 `ltc-demo` 根）：

```bash
cd ltc-demo-update-v{version}
node update.js --dry-run
node update.js --yes
```

| 参数 | 作用 |
|------|------|
| `--dry-run` | 仅显示将执行的步骤与自建子应用前缀，**不写盘** |
| `--yes` / `-y` | 跳过确认提示，执行安装（**必须**显式添加才会改文件） |
| `--strict` | Git 工作区有未提交修改时**拒绝**继续 |
| `--rollback <备份目录>` | 从 `backups/v{version}-*/` 一键回滚 |
| `--project` / `-p` | 指定项目根相对路径 |

更新包不在项目内时：

```bash
node path/to/ltc-demo-update-v{version}/update.js --project .. --yes
```

亦可使用环境变量（相对或绝对均可）：

```bash
# Windows
set LTC_PROJECT_ROOT=..
node ltc-demo-update-v{version}\update.js --yes

# macOS / Linux
LTC_PROJECT_ROOT=.. node ltc-demo-update-v{version}/update.js --yes
```

### 3.3 安装脚本自动执行项

`update.js`（即 `apply-update-package.mjs`）顺序如下：

1. **预检** — Node ≥ 18；定位项目根；版本比较（§1.4）；检测 Git 脏工作区（`--strict` 时阻断）
2. **确认** — 无 `--yes` 时仅提示计划并退出；`--dry-run` 到此结束
3. **全量备份** → `backups/v{version}-{时间戳}/` + `update-backup-manifest.json`
   - `package.json`、`package-lock.json`
   - `AIEP-WEB/src`、`build`、`scripts`、`vite.config.js`、`index.html`
   - `核心文档/框架核心文档`、`AI+产品落地`、`执行说明.txt`
   - `AIEP-SERVER/`（若存在）
4. **合并 `files/`**
   - `package.json`：依赖以包为准；`dev:*` / `build:*` / `validate:*` 脚本**保留用户已有项**
   - `AIEP-WEB/src`：覆盖框架文件；**跳过**用户独有 `apps/`、`docs/子应用文档/` 目录
   - `subApps.js`：**按 `id` 合并**，包内框架条目 + 用户自建条目
   - `build/`：**不覆盖**用户自建 `vite.<folder>.config.js`（§13.3）
   - **自动合并**自建子应用 `router/index.js` 路由块与 `App.vue` 嵌入前缀（§13.3）
5. 合并 `核心文档/AI+产品落地`、`执行说明.txt`、`AIEP-SERVER`
6. **清理废弃文档** — 删除 §2.4 所列 1.0 遗留文件（存在则删）
7. `npm install`（根目录 + `AIEP-SERVER/` 若存在）
8. **主应用验证**（§3.6）— 注册表 + `npm run build` + 产物检查；**异常则输出说明并自动回滚**
9. **子应用验证**（§3.6）— 按 `subApps.js` 逐项 `npm run build:<folder>`；**异常仅提示，不回滚**
10. 终端打印 **§12** 熟悉项目与规范清单；输出 `--rollback` 命令备查

### 3.4 更新后手动抽检（可选）

安装脚本已自动完成主/子应用构建验证。若需人工确认运行时行为，可再执行：

```bash
npm run dev
npm run validate:sub-app-registry
```

子应用构建若在安装时已提示异常，按终端「异常说明」修复后执行 `npm run build:<folder>`。

### 3.5 更新后必读（自动提示）

`update.js` 安装成功后会**在终端打印** §12 的精简清单（文档路径与命令）。完整说明见下文 **§12 更新后熟悉项目与规范执行**。

### 3.6 更新后自动验证（主应用阻断 / 子应用提示）

| 对象 | 验证项 | 失败行为 |
|------|--------|----------|
| **主应用** | `validate:sub-app-registry` | 输出 JSON `failed_items` 等异常说明 → **自动回滚** |
| **主应用** | `npm run build` + `AIEP-WEB/dist/index.html` 存在 | 输出构建日志尾部 → **自动回滚** |
| **子应用** | 对每个登记项执行 `npm run build:<folder>`（有脚本且有 `src/apps/<folder>/`） | 输出该子应用构建异常说明 → **仅提示，更新仍算成功** |
| **子应用** | 无 `build:<folder>` 脚本或源码目录缺失 | 标记「跳过」并说明原因 |

**原则**：只有主应用异常会阻断更新；子应用问题由用户按终端说明自行修复，不影响框架升级结果。

## 4. 更新包元数据

### 4.1 update-info.json（字段说明）

| 字段 | 必填 | 说明 |
|------|------|------|
| `version` | 是 | 与打包时 `package.json.version` 一致；**用于与现网比较** |
| `releaseDate` | 是 | 发布日期 ISO 日期 |
| `updateType` | 是 | 固定 `framework` |
| `updateMode` | 是 | 固定 `merge` |
| `preserveUserSubApps` | 是 | 固定 `true` |
| `minCompatibleVersion` | 否 | 现网低于此版本时拒绝升级（可选） |
| `frameworkAppFolders` | 是 | 框架子应用目录列表 |
| `obsoletePaths` | 是 | 更新后必须删除的废弃文档相对路径列表（§2.4） |
| `requirements` | 否 | Node/npm 要求 |

打包脚本自动生成，无需手写目标版本。

### 4.2 framework-meta.json（位于 files/）

供 `update.js` 判断哪些 `apps/`、`docs/子应用文档/` 子目录属于框架、可覆盖更新。

## 5. 自建子应用与合并边界

### 5.1 更新后通常无需改动（脚本自动保障）

- `src/apps/<你的应用>/` 源码 — **不删除、不覆盖**
- `src/docs/子应用文档/<你的应用>/` 文档 — **不删除、不覆盖**
- `package.json` 中已有的 `dev:<folder>`、`build:<folder>` 脚本 — **保留**
- `router/index.js` 中自建子应用路由块 — **从备份自动合并**（§13.3）
- `App.vue` 中 `SUB_APP_EMBED_PREFIXES` — **从备份自动合并**（§13.3）
- `build/vite.<folder>.config.js`（自建）— **同名不覆盖**（§13.3）

### 5.2 极少数需人工介入的情况

| 情况 | 处理 |
|------|------|
| 自建子应用路由结构非标准（脚本无法解析路由块） | 从 `backups/` 手工合并 `router/index.js` |
| 更新前注册表校验已失败（项目本身未接好） | 先按接入规范修好，再执行更新 |
| 更新后需验证自建子应用构建 | 手动执行 `npm run build:<folder>` |

> 注册表校验失败时，安装脚本会**自动回滚**，不会留下半成品。

### 5.3 禁止依赖的行为

- **不要**对 `src/` 做整目录替换（旧方案会删除 `apps/`）。
- **不要**在未备份时手动覆盖 `src/apps/`。
- **不要**省略 `--yes` 后误以为已更新（无 `--yes` 默认不写盘）。

## 6. 回滚

### 6.1 自动回滚（默认）

`validate:sub-app-registry` 或 `npm run build` 失败时，`update.js` 会**立即**从当次备份目录恢复，项目回到更新前状态。

### 6.2 手动回滚

```bash
node update.js --rollback backups/v{version}-{时间戳}
npm install
npm run build
```

备份目录含 `update-backup-manifest.json`，记录当次覆盖的路径列表与自建子应用前缀。

### 6.3 手工文件恢复（兜底）

若 `--rollback` 不可用，将 `backups/v{version}-*/` 内文件按目录结构拷回项目根，再执行 `npm install && npm run build`。

## 7. 故障处理

| 现象 | 处理 |
|------|------|
| 提示「确认继续？添加 --yes」 | 正常：先 `--dry-run` 预览，再 `--yes` 执行 |
| 提示「已是最新」/「跳过降级」 | 正常：更新包版本未高于现网，无需操作 |
| 找不到 ltc-demo 项目根 | 在项目根执行，或设置 `LTC_PROJECT_ROOT` |
| 更新中途失败并提示「已自动回滚」 | 项目已恢复；修复根因后重新 `node update.js --yes` |
| `validate:sub-app-registry` 失败（更新前） | 先按接入规范修好现网，再更新 |
| 应用中心看不到自建子应用 | 检查 `subApps.js` 是否有你的 `id`；确认更新时终端是否打印了自建前缀 |
| `npm run build` 失败 | 已自动回滚；`npm cache clean --force` 后重试；检查 Node >= 18 |
| 子应用页面空白 | 确认 `SUB_APP_EMBED_PREFIXES` 含该前缀；子应用 `base` 计算是否正确 |

## 8. 安全与分发

- 更新前**全量备份**（§3.3）；关键步骤失败**自动回滚**（§6.1）。
- 建议用户额外复制整个项目目录；有未提交 Git 修改时使用 `--strict` 或先 commit。
- 内网/U 盘分发时，可对 zip 附 `SHA256` 校验（规则见 `创建指南.md` §5.4）。
- 先在测试副本上跑通 `--dry-run` 与 `--yes`，再对主开发目录执行。

## 9. 文档与版本联动

发布新框架版本时，维护人应：

1. 更新根目录 `package.json` 的 `version`
2. 执行 `npm run create-update-package` 并在**低于新版本的**副本上验证可一键安装
3. 同步修订：`框架基线.md`、`文档对齐清单.md`、`一键更新指南.md`（用户步骤）、本文 **§12**（若接手流程有变）
4. 在 `更新内容说明.md`（可选，放项目根或随包 README）记录相对上一版的框架变更

| 安装更新包 | 包内 `update.js`（源：`apply-update-package.mjs`） | 用户执行 `node update.js` |
| Git 拉取更新 | `AIEP-WEB/src/scripts/update.js` | `npm run update` |
| 注册表校验 | `AIEP-WEB/scripts/sub-app-registry-validate.mjs` | `npm run validate:sub-app-registry` |

## 11. 总结

本方案实现 **「框架可分发、子应用零损、失败可回滚、更新后可按最新规范接手」** 的一键更新：

1. 开发者：bump `package.json.version` → `npm run create-update-package` → zip 分发  
2. 用户：`--dry-run` 预览 → `node <更新包>/update.js --yes`（**仅当包版本 > 现网版本**）  
3. 全量备份 → 合并框架 → 自动合并自建路由/嵌入前缀 → 校验门禁 → 构建主应用  
4. 校验或构建失败 → **自动回滚**；成功则保留备份供 `--rollback`  
5. 安装完成后按 **§12** 熟悉项目结构与规范，后续执行以 §12 为准  

**不再维护**本文档旧版中的内嵌整段示例代码（历史 `archiver`、扁平 `src/` 结构等）；实现以 §10 脚本为准。

## 13. 零风险保障机制

> **说明**：软件更新无法做到数学意义上的「绝对零风险」；本方案通过 **预防 → 保护 → 验证 → 回滚** 四层闭环，将可预见风险压到最低，使用户侧感知为「要么成功可用，要么原样退回」。

### 13.1 四层闭环

| 层级 | 机制 | 效果 |
|------|------|------|
| **预防** | `--dry-run` 预览；无 `--yes` 不写盘；`--strict` 拒绝 Git 脏工作区；版本不大于现网则跳过 | 避免误操作、误降级、误目录 |
| **保护** | 自建 `apps/`、文档目录不删；`subApps.js` 按 id 合并；`build/vite.<自建>.config.js` 不覆盖 | 子应用资产零损 |
| **验证** | 注册表校验 + 主应用 `build` 作为**完成门禁** | 未通过不算更新成功 |
| **回滚** | 门禁失败自动从当次 `backups/` 恢复；支持 `--rollback` | 不留半成品 |

### 13.2 全量备份范围

每次 `--yes` 安装前写入 `backups/v{version}-{时间戳}/`，并生成 `update-backup-manifest.json`：

- 根 `package.json`、`package-lock.json`
- `AIEP-WEB/` 下 `src`、`build`、`scripts`、入口 HTML/Vite 配置
- `核心文档/框架核心文档`、`AI+产品落地`、`执行说明.txt`
- `.cursor/`（`skills/`、`rules/`）
- `scripts/`（Skill 同步脚本）
- `AIEP-SERVER/`（若存在）

### 13.3 自建子应用自动合并（核心）

安装脚本在合并框架版 `router/index.js`、`App.vue` 之后，根据备份中 `subApps.js` 识别**非框架**子应用（`folder` 不在 `framework-meta.json` 的 `frameworkAppFolders` 内），并：

1. 从备份 `router/index.js` **提取**对应 `/前缀` 的顶层路由对象，追加到现网路由表（若缺失）
2. 从备份 `App.vue` **合并** `SUB_APP_EMBED_PREFIXES`，保证自建前缀不丢
3. `build/` 合并时，跳过覆盖用户独有的 `vite.<folder>.config.js`

因此 §5.2 所列「手工合并 router/App.vue」在标准接入结构下**不再需要**。

### 13.4 失败自动回滚

以下任一步骤抛错即触发回滚（恢复 §13.2 清单内全部路径）：

- 文件合并异常
- `npm install` 失败
- **主应用验证**失败（注册表 / build / 产物缺失，见 §3.6）

**不触发回滚**：子应用 `npm run build:<folder>` 失败（仅终端提示异常说明）。

终端输出：

```text
❌ 更新在「合并/安装/校验/构建」失败: ...
⏪ 正在从备份自动回滚...
✅ 已回滚至更新前状态
```

### 13.5 残余风险与建议

| 残余风险 | 建议 |
|----------|------|
| 更新包被篡改 | 分发 zip 附 SHA256；内网校验 |
| 跨大版本 API 不兼容 | 配置 `minCompatibleVersion`；逐级升级 |
| 自建子应用构建未自动执行 | 更新成功后手动 `npm run build:<folder>` |
| 对 `AI+产品落地` 模板的本地定制 | 会被框架版 merge 覆盖；重要定制应放项目实例文档目录 |
| 更新前项目注册表已损坏 | 先修现网，再更新；否则门禁失败并回滚 |
| 1.0 废弃文档仍被引用 | 更新后 §2.4 四类文件会被删除；请改读 `安装指南.md` 与 `执行说明.txt` |

### 13.6 推荐安全更新流程

```bash
# 0. 复制整个项目目录（可选但强烈建议）
# 1. 预览
node ltc-demo-update-v{version}/update.js --dry-run
# 2. 执行（Git 干净时可用 --strict）
node ltc-demo-update-v{version}/update.js --yes
# 3. 验证
npm run dev
npm run validate:sub-app-registry
npm run build:<每个自建 folder>
```
