# 系统设计规范-ant

## 一、设计理念

参考 Ant Design 设计体系，构建“确定性强、信息密度高、企业级一致性好”的业务界面。核心原则如下：

- 自然（Natural）：布局和反馈符合用户认知习惯。
- 确定（Certain）：操作结果可预期，状态表达清晰。
- 意义感（Meaningful）：信息结构清楚，视觉层级合理。
- 生长性（Growth）：支持复杂业务持续演进。

- 数字/英文：`Arial, sans-serif`
- 代码字体：`SFMono-Regular, Consolas, 'Liberation Mono', Menlo, monospace`

### 2. 字号层级

- 大标题：`38px`
- 页面标题：`30px`
- 区块标题：`24px`
- 卡片标题：`20px`
- 正文：`14px`

### 3. 字重

### 4. 行高建议

- 正文：`1.57`

## 四、组件规范

### 1. 按钮

- 默认高度：`32px`
- 小按钮高度：`24px`
- 大按钮高度：`40px`
- 默认圆角：`6px`
- 类型：主按钮、默认按钮、虚线按钮、文本按钮、链接按钮
- 状态：默认、悬停、按下、禁用、加载

### 2. 输入与表单

- 输入框高度：`32px`
- 标签字号：`14px`
- 必填标识：红色 `*`
- 校验错误：文本与边框同步警示
- 大表单采用分组与分步，避免单页过长

### 3. 表格

- 头部背景：`#FAFAFA`
- 行高建议：`48px`（可按密度切换）
- 支持固定列、排序、筛选、分页
- 空态、加载态、异常态必须可见

### 4. 卡片

- 背景：白色
- 圆角：`8px`
- 边框：`1px solid #F0F0F0`
- 阴影：轻量（仅悬停增强）

### 5. 弹窗/抽屉

- 关闭方式：右上角关闭、遮罩点击、ESC（按需）
- 弹窗推荐宽度：`520px / 720px / 960px`
- 抽屉用于“不中断上下文”的编辑场景

### 6. 反馈组件

- 成功、警告、错误、信息统一使用消息规范
- 关键失败场景需提供“重试”入口
- 长任务必须给出进度提示

## 五、布局规范

### 1. 典型后台布局

- 顶部导航：`56px ~ 64px`
- 侧边栏宽度：`200px`（展开）/ `80px`（收起）
- 内容区：12 栅格 + 间距系统

### 2. 栅格与间距

- 采用 `24` 栅格体系
- 常用间距：`8 / 12 / 16 / 24 / 32`
- 页面左右留白：`24px`（桌面），`16px`（平板），`12px`（移动）

### 3. 响应式断点

- `xs`: `<576px`
- `sm`: `>=576px`
- `md`: `>=768px`
- `lg`: `>=992px`
- `xl`: `>=1200px`
- `xxl`: `>=1600px`

## 六、交互规范

### 1. 导航交互

- 当前路由高亮必须准确
- 面包屑最多 3~4 级，避免层级过深
- 侧栏折叠时保留 Tooltip，保证可识别性

### 2. 表单交互

- 聚焦：主色边框 + 轻阴影
- 失焦：触发即时校验（按字段策略）
- 提交失败：定位首个错误字段并滚动到可视区域

### 3. 列表交互

- 查询条件变化后，分页重置到第一页
- 批量操作前需二次确认
- 导出/导入使用异步任务并展示任务状态

### 4. 真实系统交互约束（强制）

- 禁止“仅切换 UI 状态不触发业务逻辑”的伪交互，所有关键操作必须驱动真实数据变更或真实查询。
- 查询条件（筛选项、关键词、时间区间、Tab）切换后，必须触发实际查询请求并刷新结果集，不允许仅前端假过滤。
- 查询条件变更时，必须同步重置分页到第一页，并保持“条件、分页、排序”三者状态一致。
- 表单提交、开关切换、状态变更等操作必须落到真实接口；成功后回写页面，失败后给出可执行的错误提示。
- 加载中、成功、空结果、失败四类状态必须可见，且状态切换与真实请求生命周期一致。
- 关键交互需可追踪：请求参数、响应结果、错误信息在日志或调试面板中可定位，便于排障与验收。

### 5. 交互仿真化覆盖要求（强制）

- 除查询类交互外，新增、编辑、删除、批量操作、审批、发布、上下架等业务操作均需采用真实系统交互形式。
- 所有操作需具备“操作前确认 -> 请求中反馈 -> 结果回写 -> 失败可恢复”的完整链路，不允许仅展示成功提示。
- 导入/导出、上传/下载、长任务处理必须体现真实任务状态（排队、处理中、完成、失败）与进度反馈。
- 权限类交互需真实体现“可见性 + 可操作性”双重控制：无权限时不可见或禁用，并给出清晰原因说明。
- 并发与重复提交场景必须防重（按钮 loading/禁用、幂等校验），避免重复创建或重复扣减。
- 路由切换、页面刷新、返回重进后，关键业务状态应与后端状态保持一致，不允许出现前后端状态漂移。
- 异常场景必须仿真化覆盖：超时、断网、接口 4xx/5xx、部分失败，均需提供可执行补救动作（重试、撤销、回滚、联系客服）。
- 关键业务链路应提供最小可验收脚本（Given/When/Then），用于验证交互是否符合真实生产行为。

### 4. 状态规范

- 页面状态：`loading / success / empty / error`
- 按钮状态：`default / hover / active / disabled / loading`
- 异常页面：`403 / 404 / 500` 三类必须完整

## 七、数据交互与工程约束

### 1. 接口规范

- 统一响应结构：`code`、`message`、`data`、`requestId`
- 列表接口统一分页字段：`pageNo`、`pageSize`、`total`
- 时间字段统一 ISO 或毫秒时间戳，禁止混用

### 2. 错误处理

- `4xx`：业务可处理错误，给用户可读提示
- `5xx`：统一兜底提示并支持重试
- 网络异常：展示离线/超时提示，避免页面假死

### 3. 安全与审计

- 敏感操作必须二次确认并记录操作日志
- 关键接口附带操作者身份与追踪 ID
- 前端避免暴露敏感字段（如完整证件号、密钥）

## 八、主题与可配置化

- 基于 Design Token 管理样式变量，不直接散落硬编码颜色。
- 支持浅色/深色主题切换，并持久化用户偏好。
- 组件风格优先通过 Token 扩展，避免大量覆盖式 CSS。

## 九、移动端设计规范

### 1. 设计原则

- 移动优先：核心任务在单手可达范围内优先呈现。
- 简化路径：同一流程建议控制在 `3~5` 步内完成。
- 即时反馈：点击、提交、加载必须在 `100ms` 内有可见响应。
- 可恢复：关键操作提供撤销、重试或草稿保留能力。

### 2. 断点与适配

- 基准宽度：以 `375px` 设计稿为主（兼容 `360px`）。
- 常用区间：`<=360`（小屏）、`361~414`（主流手机）、`>=768`（平板）。
- 适配方式：优先使用弹性布局（Flex）和百分比宽度，避免固定像素。
- 横竖屏切换时，避免核心操作入口位置突变。

### 3. 触控与可用性

- 可点击区域最小尺寸：`44px × 44px`。
- 组件间最小安全间距：`8px`，避免误触。
- 重要按钮放置在底部可达区，避免靠近刘海/手势冲突区。
- 长列表支持“回到顶部”与吸顶筛选，减少重复滑动成本。

### 4. 布局与导航

- 页面左右留白建议：`12px ~ 16px`。
- 顶部导航高度建议：`44px ~ 56px`（含状态栏按端能力适配）。
- 底部 Tab 建议不超过 `5` 项，文案控制在 `2~4` 字。
- 多层级信息优先采用“列表 -> 详情 -> 操作”的渐进式结构。

### 5. 移动端表单规范

- 输入项按任务分组，单屏优先展示高频字段。
- 调起键盘时自动滚动聚焦字段到可视区域，避免被遮挡。
- 时间、地区、枚举优先使用选择器，减少手动输入。
- 提交按钮在可见区域常驻或就近呈现，避免用户反复上滑下滑。

### 6. 状态与反馈

- 加载超过 `300ms` 显示骨架屏或 loading，超过 `10s` 给出重试入口。
- 空态需给出下一步动作（去新增、去筛选、刷新）。
- 网络异常区分超时、断网、服务异常，并提供对应文案。
- 成功反馈建议轻提示（Toast），失败反馈需明确原因与解决建议。

### 7. 性能与弱网约束

- 首屏关键内容建议在 `2s` 内可见（4G 网络基线）。
- 图片优先使用 WebP/AVIF，并按设备分辨率下发。
- 列表分页采用懒加载或分段渲染，避免一次性渲染过多节点。
- 弱网下优先保障“查看与提交”主路径，非关键资源可延迟加载。

### 8. 移动端安全与隐私

- 涉及支付、删除、审批等高风险操作必须二次确认。
- 敏感信息默认脱敏展示，并提供按需查看机制。
- 前台缓存中避免持久化高敏数据（token、证件完整号等）。
- 截屏风险场景可按业务要求增加水印或安全提示。

## 十、实施建议

1. 先统一 Token 与基础组件（按钮、输入、表格、卡片）。
2. 再治理页面模板（列表页、详情页、表单页、大屏页）。
3. 最后补齐交互状态机与接口契约校验（加载/空态/错误态）。

## 十一、验收清单（最小集）

- 视觉：颜色、字号、间距、圆角符合 Ant 规范。
- 交互：关键链路存在明确反馈与异常兜底。
- 数据：接口字段统一、分页统一、错误码统一。
- 响应式：`xs/sm/md/lg/xl` 下无布局破坏。
- 质量：核心页面可通过构建与基础测试。
