# 过程规范文档

> **文档定位**：框架开发、维护、交接与 AI 协作的**过程治理**文档（流程步骤、责任角色、门禁术语、问题留痕）。  
> **操作 SOP**见 `核心文档/执行说明.txt`；**技术细节**见各专题指南；**文档变更联动**见 `文档联动更新规范.md`。  
> **最后对齐**：2026-06-05 · 与 `更新分发方案.md` v2 零风险机制、`主应用子应用接入规范.md` 同步。

---

## 1. 文档目的与适用范围

### 1.1 目的

1. 统一框架全生命周期过程（安装 → 开发 → 接入 → 升级 → 交活）。
2. 为 AI 助手与研发人员提供**可执行、可校验**的规则，避免口头约定与文档漂移。
3. 记录已知问题与改进措施，形成闭环。

### 1.2 适用范围

| 角色 | 使用场景 |
|------|----------|
| 框架维护人 | 发版、打更新包、修订核心文档 |
| 子应用研发 | 新建/变更子应用、接主应用、跑门禁 |
| 产品 / 评审 | 填报 SDD/PRD、过 G2/G3 |
| AI 助手 | 接手项目、改代码、写文档、执行更新 |

### 1.3 与其他文档的分工

| 文档 | 分工 |
|------|------|
| `执行说明.txt` | 接手后 **5 阶段**操作清单（偏执行） |
| `过程规范文档.md`（本文） | 过程规则、门禁、AI 约束、问题留痕（偏治理） |
| `创建指南.md` | 创建与扩展的技术实现细节 |
| `更新分发方案.md` | 离线更新包机制与零风险保障 |
| `主应用子应用接入规范.md` | 子应用接入主应用四处登记清单 |
| `AI+产品落地/` | 需求 / 设计 / 门禁模板与 SOP |

## 2. 框架生命周期过程

### 2.1 新建与安装

| 步骤 | 动作 | 权威文档 |
|------|------|----------|
| 1 | 检查 Node ≥ 18、Git | `安装指南.md` |
| 2 | `npm install`（根 + `AIEP-SERVER` 若需要） | `安装指南.md` |
| 3 | `npm run dev` / `npm run build` | `主应用打包指南.md` |
| 4 | 阅读框架结构与规范 | `创建指南.md`、`执行说明.txt` §文档优先级 |

### 2.2 子应用开发与接入

严格按 `主应用子应用接入规范.md` **Checklist**，不得跳步：

1. **文档**（四段式目录，参照 `sample-app`）：
   - 根路径：`AIEP-WEB/src/docs/子应用文档/<appCode>/`
   - 必须先创建四个阶段目录，再从 `AI+产品落地/02-子应用通用模板/` 复制模板到对应目录：
     - `01-需求与设计/` — 需求说明书、PRD、SDD（md+json）、页面-路由-接口-数据表映射表
     - `02-研发与测试/` — 详细设计、数据库设计、验收脚本-GWT、G3 任务拆解等
     - `03-发布与复盘/` — 应用操作说明书
     - `04-AI治理与审计/` — `gate-config.json`、SDD 校验清单、G2-G3 门禁检查
   - **禁止**在 `<appCode>/` 根目录平铺文档（1.0 扁平结构已废止）
   - 完整文件清单见 `AI+产品落地/03-其他过程文档-参考/调整方案.md` §七
2. 代码：`src/apps/<folder>/` + `build/vite.<folder>.config.js`
3. 主应用四处：`subApps.js`、`router/index.js`、`App.vue`、`package.json` 脚本
4. 交活前：`npm run validate:sub-app-registry`；已接 SDD 则 `npm run validate:sdd -- --app <appCode> --gate G2`

### 2.3 框架升级（离线更新包）

| 步骤 | 命令 / 动作 | 说明 |
|------|-------------|------|
| 1 | `node <更新包>/update.js --dry-run` | 预览，不写盘 |
| 2 | `node <更新包>/update.js --yes` | 确认安装 |
| 3 | 阅读终端 §12 指引 | 熟悉项目与规范 |

**零风险规则**（详见 `更新分发方案.md` §13）：

- 全量备份 → 合并 → **主应用验证**（注册表 + build）→ 失败**自动回滚**
- **子应用 build 验证**失败 → 仅输出异常说明，**不回滚**
- 自动删除 1.0 废弃文档（§2.4 下表）
- 勿与 `npm run update`（Git pull）混用

### 2.4 1.0 废弃文档（更新后必须删除）

|----------|------|
| `环境安装指南.md` | `安装指南.md` |

| `系统设计规范.md`（旧版独立文档） | `创建指南.md` §3.3.4 + `安装指南.md` |

### 2.5 产品 / 门禁过程

| 阶段 | 入口 |
|------|------|

## 3. 执行后校验（强制）

任何「创建 / 修改 / 更新」操作完成后，按场景执行：

### 3.1 主应用变更

```bash
npm run validate:sub-app-registry
npm run build
```

未通过视为**阻塞**（更新场景下由 `update.js` 自动回滚）。

### 3.2 子应用变更

```bash
npm run dev:<folder>          # 开发验证
npm run build:<folder>        # 构建验证
npm run validate:sub-app-registry   # 若改了接入三件套
npm run validate:sdd -- --app <appCode> --gate G2   # 若适用
```

子应用 build 失败在**框架更新**场景下仅提示；在**日常开发**场景下由研发自行修复后再提交。

### 3.3 文档变更

按 `文档联动更新规范.md` 执行三步：

1. 影响分析  
2. 联动更新（见该规范 §4 矩阵）  
3. 术语一致性检查（关卡名、模板路径、脚本命令）

未完成联动且无暂缓说明 → 状态标记 `blocked`。

## 5. 问题记录与改进（留痕）

| 问题描述 | 发现阶段 | 影响 | 改进方法 | 状态 |
|----------|----------|------|----------|------|
| 1.0 扁平文档与 2.0 目录并存，接手人误读旧指南 | 框架升级 | 安装/规范理解偏差 | 更新包强制删除 §2.4 四类废弃文档；§12 指引 | 已落地 |
| 应用中心 / 首页指标多处硬编码子应用 | 1.0 迁移 | 漏展示、计数不准 | 统一 `subApps.js` + `validate:sub-app-registry` | 已落地 |
| 离线更新覆盖 `router`/`App.vue` 导致自建子应用不可见 | 框架升级 | 子应用「丢失」 | 自动合并自建路由与嵌入前缀；失败回滚 | 已落地 |
| `npm run update` 与离线包混用 | 协作升级 | 行为不一致 | 文档明确双路径勿混用（`更新分发方案.md` §1.2） | 已落地 |
| 文档单点更新导致术语漂移 | 日常维护 | G2/G3 口径不一 | `文档联动更新规范.md` 联动矩阵 + `blocked` | 持续执行 |
| 更新成功但子应用 build 失败 | 框架升级 | 子应用不可用 | 主应用阻断 / 子应用仅提示（§3.6） | 已落地 |

**新增问题**：按上表格式追加一行，并在同次或下次发版时写入改进措施与关联文档。

## 6. AI 助手规则

### 6.1 通用原则

1. **准确性**：以仓库代码与 §6.2 真值为准，不凭记忆或过期文档片段。
2. **完整性**：给出可执行命令与路径，不只给结论。
3. **一致性**：命名、门禁术语与 `AI+产品落地` 模板一致。
4. **最小改动**：只改任务相关文件，不顺手重构。

### 6.2 单一事实来源

| 关注点 | 真值 |
|--------|------|
| 子应用登记 | `AIEP-WEB/src/config/subApps.js` |
| 路由 / 嵌入 | `router/index.js`、`App.vue` → `SUB_APP_EMBED_PREFIXES` |
| 脚本命令 | 根目录 `package.json` → `scripts` |
| 项目实例文档 | `AIEP-WEB/src/docs/子应用文档/<appCode>/`（四段目录：`01-需求与设计` … `04-AI治理与审计`） |
| 目录规范样例 | `AIEP-WEB/src/docs/子应用文档/sample-app/` |
| 填报模板 | `核心文档/AI+产品落地/02-子应用通用模板/` |
| 框架升级 | `更新分发方案.md`、`apply-update-package.mjs` |

### 6.3 场景规则

| 场景 | 必须 |
|------|------|
| 新建子应用 | 走接入规范 Checklist；文档按四段目录从 `02-子应用通用模板` 复制；跑 `validate:sub-app-registry` |
| 改框架 / 发更新包 | 读 `更新分发方案.md`；bump `package.json.version` 后 `create-update-package` |
| 写需求 / PRD / SDD | 只从 `02-子应用通用模板` 复制；不写散落旧模板 |
| 文档变更 | 查 `文档联动更新规范.md` 矩阵，同步关联文档 |
| 规范冲突 | 代码 + `文档对齐清单.md` > 旧版记忆；治理版 > 执行版争议时 |

### 6.4 禁止行为

- 在 `AppCenter.vue` / `Dashboard.vue` 硬编码子应用列表或页面数  
- 对 `src/` 整目录替换（会删除用户 `apps/`）  
- 未 `--yes` 即认为离线更新已完成  
- 跳过 `validate:sub-app-registry` 声称「已接入主应用」  
- 文档变更不联动却标记关卡通过  
- 在 `子应用文档/<appCode>/` 根目录平铺需求/PRD/SDD 等文件（须落入四段目录）

## 7. 审查流程

### 7.1 触发时机

- 完成子应用接入、框架升级、SDD 阶段交付、核心文档批量修订后。

### 7.2 审查清单（可复制）

```text
过程审查：
- [ ] 主应用：validate:sub-app-registry 通过
- [ ] 主应用：npm run build 通过
- [ ] 子应用：npm run build:<folder> 通过（适用时）
- [ ] SDD：validate:sdd 通过（适用时）
- [ ] 文档：联动矩阵关联项已更新或已标注暂缓
- [ ] 更新：若刚执行 update.js，终端无「主应用验证失败」；子应用异常已记录
- [ ] 废弃：项目内无 §2.4 四类旧文档残留
- [ ] 问题表：新发现问题已写入 §5
```

### 7.3 审查步骤

1. **划定范围** — 代码 / 文档 / 更新包影响面  
2. **跑校验脚本** — §3.1～§3.2  
3. **对照真值** — §6.2 与 `文档对齐清单.md`  
4. **记录与闭环** — §5 问题表 + 修订记录
